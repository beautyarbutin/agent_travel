## 先统一项目口径

### 你可以先背这一段 30 秒总述

这个项目不是简单的聊天 demo，而是一个 `Router + 专家 Agent + MCP 工具层 + 结构化记忆 + Hybrid RAG + LoRA + 离线评测` 的多智能体旅游助手。  
我主要做了四件事：第一，把系统从广播式回复改成 `travel_router` 集中路由；第二，把天气、景点、路线能力做成 MCP 工具，降低幻觉；第三，做了本地 Hybrid RAG，知识库扩到 `15723` 条；第四，基于 LLaMA-Factory 做了 `router_lora` 和 `spot_lora`，其中 Router 这条线效果最明显，完整链路成功率从 `20.59%` 提升到 `85.29%`。

### 面试里建议优先说的硬指标

- 知识库规模：`15723` 条
- Router 训练数据：`833` 条
- Spot 训练数据：`8802` 条
- Router benchmark：`34` 条
- Spot internal benchmark：`80` 条
- CrossWOZ benchmark：`150` 条
- RAG benchmark：`198` 条
- Router 完整链路成功率：`20.59% -> 85.29%`
- Router turn exact：`48.81% -> 94.05%`
- Router argument exact：`19.57% -> 89.13%`

### 面试里一定要稳着说的 4 个点

1. `spot_lora` 不是“全面提升”，更准确的说法是：**风格更像旅游助手，但严格事实槽位指标下降**。公开 CrossWOZ 原始对比口径里，`Strict success 98.00% -> 78.00%`，`Fact coverage 99.00% -> 89.11%`，`ROUGE-L F1 2.66% -> 27.36%`。
2. 当前 RAG 不是“标准 BM25 + rerank”，而是 `FAISS + BM25-like + handwritten bonus` 的工程化混合排序。
3. 当前没有独立 cross-encoder reranker。
4. RAG 现成报告的主结果来自 `2026-03-13` 的 `15240` 条旧库版本；当前运行库已经扩到 `15723` 条，所以要说成“**报告基于旧版本，运行库已继续扩充**”。

## 1. 怎么解决幻觉，输出慢怎么解决？

### 【面试当场如何回答】

我会把幻觉分成两类：一类是**事实型幻觉**，比如天气、路线、地址这类本来应该查工具却被模型编出来；另一类是**行为型幻觉**，比如 Router 本来只该分发，却越权自己查天气。  
我的做法不是单靠提示词，而是四层约束：高风险事实任务强制走工具，Router 和专家 Agent 做权限隔离，景点场景用 `POI + 本地 RAG` 做 grounding，再用评测和 badcase 反推问题。  
输出慢主要来自长上下文、外部 API、RAG 检索和多 Agent 串行编排。我会从缩短上下文、POI-first、缓存、减少不必要链路这几个方向优化。

### 【详细解释】

- 事实型幻觉：天气、路线、景点地址、路线耗时这类问题，本质上应该先拿外部真实结果。
- 行为型幻觉：Router 不该直接调用业务工具，专家 Agent 不该跨域回答，这类错本质上是“动作错”。
- 当前项目压幻觉的核心做法：
  - 天气问题强制走 `get_weather` / `get_weather_forecast`
  - 路线问题强制走 `get_driving_route`
  - 景点问题不是只靠模型回忆，而是 `高德 POI + 本地 RAG`
  - `travel_router` 只做记忆读取、指代消解、意图分发和状态保存
- 当前项目的慢，近似可拆成：

`T_total ≈ T_prefill + T_decode + T_tool + T_network + T_retrieval + T_orchestration`

| 慢的来源 | 当前项目里的表现 | 优化思路 |
| --- | --- | --- |
| 上下文太长 | 历史消息多，prefill 变慢 | 用结构化记忆替代整段原文重放 |
| 工具网络调用 | 天气、POI 都有外部请求 | 做短时缓存、减少重复查询 |
| RAG 检索 | 加载 embedding / FAISS / 排序 | 懒加载、缓存热门查询 |
| 多 Agent 编排 | Router 先分发，专家再执行 | 简单任务走最短路径，少走无效链路 |

## 2. Agent 间数据怎么流动？travel_router 如何把天气问题交给 weather_agent？

### 【面试当场如何回答】

当前项目真实形态不是广播抢答，而是 `Router + Direct Message 专家 Agent`。  
用户先在公共频道发消息，`travel_router` 监听公共频道，先 `get_context`，再做指代消解和意图识别，然后通过 `send_direct_message(target_agent_id="weather_agent", text=...)` 把任务交给 `weather_agent`。  
`weather_agent` 只监听来自 Router 的 direct message，收到后再 `get_context -> get_weather -> send_channel_message`，把结果回到公共频道。

### 【详细解释】

| 环节 | 谁负责 | 做什么 |
| --- | --- | --- |
| 公共频道入口 | `travel_router` | 监听 `thread.channel_message.notification` 和 `thread.reply.notification` |
| 读状态 | `travel_router` | `get_context()` |
| 任务分发 | `travel_router` | `send_direct_message()` 给目标 Agent |
| 专家执行 | `weather_agent` / `spot_agent` / `plan_agent` | 调用各自业务工具 |
| 最终回显 | 专家 Agent | `send_channel_message(channel="general", text=...)` |

天气问题完整链路可以直接这么说：

```text
用户在 general 发问
-> travel_router 收到消息
-> get_context 读取当前记忆
-> 判断是天气意图
-> send_direct_message 给 weather_agent
-> 视情况 save_context(current_city=...)
-> weather_agent 收到 direct_message
-> get_context 补全上下文
-> get_weather(city=...)
-> send_channel_message 回 general
```

这里真正流动的不是一种数据，而是三类：

1. 自然语言任务数据：用户原始问题和 Router 转发后的文本。
2. 结构化状态数据：`current_city`、`current_spot`、`preferences` 等。
3. 工具返回的外部事实数据：天气结果、POI、路线结果。

当前实现的一个关键点是**发送者过滤**。Router 会忽略来自其他 Agent 的消息，避免出现“Agent 听到 Agent 的回复又重新路由”的回声循环。

## 3. LoRA 和改提示词有什么区别？怎么评估？是不是只学了原理？超参数怎么定？为什么 checkpoint 不是 final？

### 【面试当场如何回答】

Prompt 优化和 LoRA 微调改的是两件事：Prompt 改的是输入条件和规则约束，LoRA 改的是模型参数里的行为分布。  
这个项目里我不是只学了原理，而是真的做了两套 LoRA：`router_lora` 和 `spot_lora`。其中 Router 这条线效果最明确，完整链路成功率从 `20.59%` 提升到 `85.29%`。  
checkpoint 也不是看 final 就行，我是结合 `eval_loss + 下游 benchmark` 选的。Router 最后选的是中间的 `checkpoint-60`，因为它在真实工具调用链上最好。

### 【详细解释】

| 方案 | 改什么 | 优点 | 局限 |
| --- | --- | --- | --- |
| Prompt 优化 | 输入条件、few-shot、规则 | 快、便宜、透明 | 容易不稳定，规则不一定被真正学住 |
| LoRA | 低秩参数增量 | 能把角色行为学进参数，成本低于全参 | 容量有限，数据和指标不对齐时也会学偏 |
| 全参微调 | 全模型参数 | 理论容量最大 | 成本高、易破坏通用能力、工程维护重 |

LoRA 核心公式可以记这句：

`W' = W0 + (alpha / r) * B A`

当前项目的两条 LoRA：

- `router_lora`
  - 目标：意图识别、路由、参数构造、记忆读写顺序
  - 关键结果：
    - `Tool-name accuracy 51.19% -> 100.00%`
    - `Argument exact match 19.57% -> 89.13%`
    - `Turn exact match 48.81% -> 94.05%`
    - `Sample full-chain success 20.59% -> 85.29%`
- `spot_lora`
  - 目标：景点回答更像旅游助手
  - 实际结果：风格增强明显，但 strict factual 指标下降

当前可复核超参数：

| 项目 | `router_lora` | `spot_lora` |
| --- | --- | --- |
| base model | `Qwen3.5-4B` | `Qwen3.5-4B` |
| `lora_rank` | `8` | `8` |
| `lora_alpha` | `16` | `16` |
| `lora_dropout` | `0.05` | `0.05` |
| `learning_rate` | `5e-5` | `3e-5` |
| `cutoff_len` | `2048` | `1536` |
| `num_train_epochs` | `1.5` | `1.0` |
| `gradient_accumulation_steps` | `8` | `8` |
| `val_size` | `0.1` | `0.05` |
| template | `qwen3_5_nothink` | `qwen3_5_nothink` |

为什么 `checkpoint` 不一定选 final：

- 训练损失最低，不等于下游业务指标最好。
- Router 关心的是“工具选对没有、参数填对没有、整条链对没有”，这和 token-level loss 不是一个目标。
- 所以 Router 最终选中间 checkpoint，本质上是**以下游行为指标优先**，不是盲信 final。

## 4. 为什么 rank 常设成 8、16？alpha 为什么常是 rank 的两倍？target module 有什么作用？q/k/v/o 有什么区别？

### 【面试当场如何回答】

`rank` 决定 LoRA 更新子空间的容量，越大表达能力越强，但显存和训练成本也越高。  
`alpha` 是缩放系数，真正起作用的是 `alpha / rank`。很多工程里常设成 `2` 左右，比如 `r=8, alpha=16`，这是经验上比较稳，不是硬性定律。  
`target module` 决定你到底改模型的哪一部分能力。加在 `q/k/v/o` 上，本质上是在改注意力里的“看什么、怎么匹配、取回什么、怎么汇总”。

### 【详细解释】

`rank = r` 会直接影响新增参数量，近似可写成：

`Params_LoRA ≈ r * (d_in + d_out)`

所以：

- `r` 太小：容量不够，容易欠拟合
- `r` 适中：能学到角色行为，成本可控
- `r` 太大：不一定继续提升，还可能把风格学过头

为什么很多项目从 `8`、`16` 开始：

- 不是玄学
- 是容量、泛化、显存之间的工程折中
- 当前项目任务偏“角色行为和格式”，不是重新学世界知识，所以 `r=8` 足够先验证收益

| 参数 | 作用 | 你可以怎么说 |
| --- | --- | --- |
| `rank` | 更新容量 | 决定 LoRA 能改多大自由度 |
| `alpha` | 更新缩放 | 真正有效的是 `alpha / rank` |
| `dropout` | 正则 | 防止适配器学太死 |
| `target module` | 改哪部分层 | 决定模型哪部分能力被改动 |

`q/k/v/o` 可以这样记：

| 模块 | 直观含义 | 改它意味着什么 |
| --- | --- | --- |
| `q_proj` | 我想关注什么 | 改关注点、提问方式 |
| `k_proj` | 别人怎么被我匹配到 | 改可匹配性、索引方式 |
| `v_proj` | 我取回什么内容 | 改被聚合的信息表示 |
| `o_proj` | 多头结果怎么合并回主干 | 改汇总后的输出组织 |

当前项目里 `lora_target = all`，说明不只是改 `q/k/v/o`，还改了更多线性层。这样做的原因是任务不只是注意力重排，还包括输出形式和角色表达。

## 5. 为什么加了个东西，模型效果反而变差？Spot LoRA 为什么有指标下降？

### 【面试当场如何回答】

我不会把它说成“训练失败”，更准确的说法是：`spot_lora` 出现了**结构性变化**。  
公开评测里它的 `ROUGE-L F1` 明显上升，说明更像旅游助手；但 `Strict success` 和 `Fact coverage` 下降，说明它更会说了，但不一定更像严格槽位抽取器。  
根本原因是训练目标、数据分布和评测指标没有完全对齐。后续改进方向不是盲目继续训，而是把“事实抽取”和“自然表达”拆开。

### 【详细解释】

原始公开 CrossWOZ 对比口径：

| 指标 | Base | Spot LoRA | 结论 |
| --- | ---: | ---: | --- |
| Strict success | `98.00%` | `78.00%` | 下降 |
| Fact coverage | `99.00%` | `89.11%` | 下降 |
| ROUGE-L F1 | `2.66%` | `27.36%` | 明显上升 |

这说明它不是“全面变差”，而是：

1. 更像自然的旅游助手了。
2. 但对 benchmark 关心的槽位边界没那么稳定。

主要原因：

- SFT 优化的是 token 概率，不是 strict slot accuracy。
- `spot` 数据更偏景点/文旅问答和攻略表达，不是纯字段抽取数据。
- 模型同时被要求做三件事：读检索上下文、抽准事实、说得自然，这三件事不完全同向。

你可以把问题总结成一句：

**这更像“风格增强 + 结构约束不足”，而不是简单一句过拟合或欠拟合就能解释。**

更好的改法：

| 改进方向 | 作用 |
| --- | --- |
| 先结构化再生成 | 先出字段，再渲染自然语言 |
| 补 strict-format 数据 | 强化门票、时长、名称等关键槽位 |
| 明确字段覆盖模板 | 没信息也要写“未查到”，不要省略 |
| facts / style 分开做 adapter | 避免一个 adapter 同时扛两类目标 |

## 6. RAG 你怎么实现的？怎么排查？为什么改了好几版？chunk 怎么切？有没有 rerank？怎么评估？

### 【面试当场如何回答】

这个项目的 RAG 不是单一向量库，而是 `高德 POI + 本地 FAISS + BM25-like` 的 Hybrid RAG。  
本地库目前是 `15723` 条，embedding 用 `BAAI/bge-small-zh-v1.5`，索引用 `FAISS IndexFlatIP`，查询时再叠加 BM25-like 词法分和手写攻略 bonus。  
当前 chunk 不是高级自适应切分，而是**一条结构化景点记录就是一个检索单元**；也没有独立 cross-encoder rerank。排查时我按 `query 解析 -> 召回 -> 排序 -> 生成 -> 数据源` 五层走。

### 【详细解释】

#### 6.1 当前 RAG 的两层 Hybrid

| 层级 | 混合了什么 |
| --- | --- |
| 系统层 | 本地 RAG + 高德 POI API |
| 本地检索层 | dense 检索 + BM25-like lexical + handwritten bonus |

#### 6.2 当前知识库和向量栈

- 当前运行库：`15723` 条
- 主要来源：
  - Kaggle/去哪儿网：`14581`
  - CrossWOZ：`465`
  - 独家手写攻略：`347`
  - China312：`312`
  - 香港旅游 QA：`18`
- embedding：`BAAI/bge-small-zh-v1.5`
- index：`FAISS IndexFlatIP`
- 文档元数据：`storage/doecment.json`

#### 6.3 当前检索是怎么做的

可直接背这个公式：

`score(d, q) = 5 * dense_score + bm25_like_score + handwritten_bonus`

其中：

- `dense_score`：FAISS top-5 向量相似度
- `bm25_like_score`：查询时现算的启发式词法分
- `handwritten_bonus`：命中独家手写攻略且有词法匹配时加权

#### 6.4 embedding 和 one-hot 的区别

| 表示方式 | 特点 | 适不适合当前项目 |
| --- | --- | --- |
| one-hot / bag-of-words | 稀疏、词面匹配强、无语义相似性 | 不适合旅游问法多变场景 |
| embedding | 稠密、能表达语义接近 | 适合做近邻检索和语义召回 |

#### 6.5 chunk 怎么切

当前最稳妥的说法是：

- 一条结构化景点/攻略记录就是一个 chunk
- 不是按 token 滑窗切
- 也不是语义自适应切分

优点：

- 业务语义完整
- 景点名、城市名、标签不会被切碎
- 好解释

缺点：

- 长内容的局部定位能力有限
- 精确景点 deep match 还有提升空间

#### 6.6 有没有 rerank

严格说：

- **没有独立 cross-encoder reranker**
- 当前是融合排序，不是标准“bi-encoder 召回 -> cross-encoder 复排”

#### 6.7 为什么改了好几版

主要是因为 badcase 不止一个来源：

1. 城市/地区解析错误
2. 城市级泛推荐误用本地 RAG，导致不够像真实 POI 推荐
3. 知识库覆盖不足，需要继续补数据源
4. 手写攻略价值高，需要在排序里显式抬权

#### 6.8 怎么排查

我会按五层定位：

1. query 有没有被错改写
2. topK 里有没有召回到目标
3. 好结果是否被排序压下去
4. 检索到了但生成没说对
5. 数据源本身有没有

#### 6.9 怎么评估

当前离线检索指标是：

- `Hit@1 / Hit@3 / Hit@5`
- `MRR`
- 严格 Doc-ID 口径
- 宽松实体级口径

现成报告主结果基于 `15240` 条旧库版本：

| 口径 | Hit@1 | Hit@3 | Hit@5 | MRR |
| --- | ---: | ---: | ---: | ---: |
| 严格 Doc-ID | `35.9%` | `36.9%` | `36.9%` | `0.3636` |
| 宽松实体级 | `42.9%` | `43.9%` | `43.9%` | `0.4343` |

分类上最值得说的结论：

- 手写攻略很强
- 城市级泛问法相对较稳
- 精确景点 strict 检索还是短板

## 7. 如果 JSON 格式输出错了怎么办？

### 【面试当场如何回答】

我不会只答“让模型重试一次”。更工程化的做法是：**优先避免自由 JSON，改用 Function Calling**。  
如果必须输出 JSON，我会做四层防护：先定义 schema，再做解析和校验，失败时做 repair，repair 后还失败就重试或者回退。  
当前项目在线链路已经尽量不靠自由 JSON 直接驱动高风险动作，而是优先走 Function Calling 和 MCP。

### 【详细解释】

自由 JSON 常见问题：

- 少逗号
- 引号不匹配
- 键名漂移
- 字段漏填
- 中文解释混进结构体

更稳的链路是：

`生成 -> 解析 -> 校验 -> 修复/重试 -> 决策执行`

而不是：

`生成 -> 看起来像 JSON -> 直接执行`

| 层 | 做法 |
| --- | --- |
| 第 1 层 | 定义 schema，明确必填字段、类型、枚举 |
| 第 2 层 | `json.loads` 和 schema validate 分开做 |
| 第 3 层 | repair 常见脏格式 |
| 第 4 层 | repair 失败就重试或回退，不执行高风险动作 |

当前项目最值得说的一句是：

**在线链路优先用 Function Calling / MCP，离线清洗阶段才允许用 repair 脚本修脏 JSON。**

## 8. 为什么这个项目算多智能体？为什么不用一个大模型直接做完？多 Agent 四种范式是什么？和单 Agent 比有什么优势？

### 【面试当场如何回答】

这个项目之所以算多智能体，不是因为写了几个 YAML，而是因为它真实存在**角色拆分、权限边界、消息协作**。  
当前是典型的 `Router + 专家 Agent` 范式：`travel_router` 做总控，`weather/spot/plan` 做专责执行。  
相比单 Agent，多 Agent 的价值在于边界更清晰、权限更容易控、评测更容易拆、后续也更容易做角色级微调和扩展。

### 【详细解释】

当前项目满足多 Agent 的三个关键条件：

1. 角色分工明确
2. 权限边界明确
3. 通过消息协作完成任务

| 角色 | 当前职责 |
| --- | --- |
| `travel_router` | 读记忆、指代消解、意图识别、分发、保存状态 |
| `weather_agent` | 天气查询 |
| `spot_agent` | 景点/攻略查询 |
| `plan_agent` | 路线和行程规划 |

为什么不直接一个大模型全做：

| 单 Agent 痛点 | 多 Agent 怎么解决 |
| --- | --- |
| Prompt 太重、边界模糊 | 每个 Agent 的规则更聚焦 |
| 工具权限过大，容易越权 | Router 和专家权限分离 |
| 问题定位困难 | 能拆开评估路由、检索、生成 |
| 角色级微调困难 | Router/Spot 可以分开训练 LoRA |

多 Agent 四种常见工程范式，你可以这样答：

1. `Router / Hub-and-Spoke`
2. `Planner-Executor`
3. `Blackboard / Shared Memory`
4. `Debate / Negotiation / Decentralized`

当前项目最主要是第一种，带一点共享记忆味道，但不是去中心化协商。

代价也要主动提：

- 链路更长，时延更高
- 编排更复杂
- 需要处理回声循环和状态同步

## 9. LoRA 数据集是怎么制作的？有没有 badcase 分析？

### 【面试当场如何回答】

我做 LoRA 不是找个数据集直接丢进去跑。Router 和 Spot 两条线的数据目标完全不同。  
Router 数据本质上是**工具调用轨迹数据**，重点是学会什么时候读记忆、什么时候分发、参数怎么补；Spot 数据更偏景点问答和攻略表达。  
训练之外，我还做了独立评测集和 badcase 分析。Router 的 badcase 主要从“动作经常错”收敛到“少数字段没补全”；Spot 的 badcase 则暴露出风格化 SFT 和 strict 槽位评测之间的矛盾。

### 【详细解释】

#### 9.1 Router 数据怎么做

- 训练目标：工具调用和动作链，而不是自由作文
- 核心流程：
  - 先生成 ShareGPT 风格样本
  - 包含 `human / gpt / tool_calls / observation`
  - 再转换成 LLaMA-Factory 更适合训练的 `function_call / observation`

关键文件：

- `data/lora_ready/01_tool_calling.json`
- `data/lora_ready/01_tool_calling_lf.json`
- 当前样本量：`833`

#### 9.2 Spot 数据怎么做

- 文件：`data/lora_ready/02_spot_cultour_clean.json`
- 当前样本量：`8802`
- 目标：景点/文旅问答和攻略表达，不是纯工具调用

#### 9.3 评测集为什么要独立

| 数据 | 作用 |
| --- | --- |
| Router train | 训练行为模式 |
| Router eval | 单独验证工具链是否稳定 |
| Spot internal | 看项目真实场景效果 |
| CrossWOZ | 看公开 benchmark 表现 |

#### 9.4 badcase 怎么看

我一般分三层：

1. 动作错没错
2. 事实错没错
3. 格式错没错

Router 的典型 badcase：

- 该 `send_direct_message` 却直接调业务工具
- `save_context` 字段漏填
- 多轮指代场景参数补不全

Spot 的典型 badcase：

- 游玩时长不稳
- 门票/预算类信息覆盖不完整
- 景点名没显式点出
- 更会写描述，但不更贴评测脚本想要的槽位格式

## 10. 这个项目的提示词工程/优化做了什么工作？

### 【面试当场如何回答】

我会把这题答成：这个项目的 Prompt 工程，不是“写几句漂亮话”，而是在给 Agent 设计**行为协议**。  
Router 的 prompt 重点是权限和工作流约束，比如先 `get_context`、做指代消解、再分发、必要时 `save_context`；Weather 的重点是必须走真实天气工具；Spot 的重点是把 POI 和本地 RAG 的使用边界写清楚；Plan 的重点是路线和行程专责化。  
Prompt 负责写制度，LoRA 负责把制度学稳。

### 【详细解释】

| Agent | Prompt 优化重点 |
| --- | --- |
| Router | 先读记忆、做指代消解、只能分发、不能越权 |
| Weather | 必须走真实天气工具，只处理天气域 |

- 强工作流约束

这类 Prompt 的核心作用是让 Agent 更像在执行协议，而不是即兴发挥。

可以直接背这句：

**Prompt 决定运行时规则，LoRA 决定模型对这些规则的内化程度。**

- `current_city`

- `departure_time`
- `trip_days`

## 16. Function Calling 是怎么回事？你项目里具体怎么体现？

### 【面试当场如何回答】

Function Calling 的本质不是“模型会写函数”，而是让模型从“说话”转成“做动作”。  
在这个项目里，Router、Weather、Spot、Plan 都在做不同形式的函数调用决策。比如 Router 要决定是 `get_context`、`send_direct_message` 还是 `save_context`；Weather 要决定什么时候调 `get_weather`。  
MCP 负责把工具接进来，Function Calling 负责让模型选对工具和参数。

### 【详细解释】

一个最简单的对比：

| 方式 | 模型做什么 |
| --- | --- |
| 普通文本生成 | 直接给你一句自然语言答案 |
| Function Calling | 先输出“该调哪个函数、带什么参数” |

可以把它理解成：

- 文本生成：`y = p_theta(. | x)`
- 动作预测：`a = policy_theta(x)`

当前项目里的体现：

| Agent | 核心动作链 |
| --- | --- |
| Router | `get_context -> send_direct_message / send_channel_message -> save_context` |
| Weather | `get_context -> get_weather / get_weather_forecast -> send_channel_message` |
| Spot | `get_context -> search_spots / search_local_knowledge / search_combined -> send_channel_message` |
| Plan | `get_context -> get_driving_route -> send_channel_message` |

MCP 和 Function Calling 的关系：

| 概念 | 作用 |
| --- | --- |
| MCP | 工具接入层，把能力暴露成标准接口 |
| Function Calling | 模型动作层，决定调哪个工具和传什么参数 |

一句话记忆：

**MCP 是把工具摆上桌，Function Calling 是让模型学会什么时候拿哪个工具。**

## 17. SFT 这种垂域微调的意义和价值在哪里？

### 【面试当场如何回答】

SFT 的价值不在于替代知识库和工具，而在于把**角色、流程、格式、风格**更稳定地学进模型参数。  
在这个项目里，Router LoRA 的明显提升就是最好的证据，它让模型从“知道规则”变成“更稳定执行规则”；Spot LoRA 也说明了另一面，就是如果数据和指标没对齐，SFT 也可能把模型往“更会说但不一定更准”的方向推。  
所以工具、RAG、SFT 不是替代关系，而是分工关系。

### 【详细解释】

SFT 最稳妥的一句话定义：

**在给定输入条件下，提高参考输出序列的条件概率。**

工程上它最擅长优化的是：

1. 输出格式
2. 角色风格
3. 任务流程
4. 领域表达习惯

工具、RAG、SFT 的分工：

| 组件 | 解决什么问题 |
| --- | --- |
| 工具 | 实时外部事实获取 |
| RAG | 外部知识检索补充 |
| SFT / LoRA | 稳定使用这些能力的行为模式 |

所以这个项目里最有价值的 SFT，不是让模型“背更多旅游知识”，而是让它更稳地：

- 先读记忆
- 正确路由
- 正确调工具
- 用目标风格组织结果

## 18. 讲解一下 Attention 机制？还有别的注意力方式吗？K、V 维度不同怎么办？padding、裁剪、维度变换怎么处理？

### 【面试当场如何回答】

标准 Attention 的核心是：用 `Q` 和 `K` 做匹配，用匹配权重去加权汇总 `V`。公式是  
`Attention(Q, K, V) = softmax(QK^T / sqrt(dk)) V`。  
工程上最重要的是理解四件事：为什么要除 `sqrt(dk)`、padding 要做 mask、K 和 V 在序列维上必须对齐、Q 和 K 的最后一维必须映射到同一匹配空间。  
别的注意力方式可以提加性注意力、稀疏注意力、线性注意力、MQA/GQA。

### 【详细解释】

#### 18.1 标准公式

`Q = XWQ`  
`K = XWK`  
`V = XWV`

`Attention(Q, K, V) = softmax(QK^T / sqrt(dk)) V`

为什么除 `sqrt(dk)`：

- `dk` 大时，点积方差会变大
- softmax 容易过尖
- 训练不稳定

#### 18.2 Multi-Head Attention

- 不同 head 看不同关系
- 最后 `Concat` 后再乘 `WO`

#### 18.3 其他注意力方式

| 类型 | 特点 |
| --- | --- |
| 加性注意力 | 早期 seq2seq 常见，用小网络打分 |
| 余弦注意力 | 用余弦相似度做匹配 |
| 稀疏注意力 | 只算局部或稀疏连接，降复杂度 |
| 线性注意力 | 通过核技巧近似降低复杂度 |
| MQA / GQA | 共享或分组共享 K/V，提升推理效率 |

#### 18.4 K、V 维度不同怎么办

要区分两种“不同”：

1. 序列长度不同：在 cross-attention 里很正常，但 `K` 和 `V` 的序列长度要一致。
2. 特征维度不同：真正点积时要求 `Q` 和 `K` 的最后一维一致，所以通常先线性投影到同一个 `dk`；`V` 的 value 维 `dv` 可以不同。

#### 18.5 padding、裁剪、维度变换

- padding：加 mask，让 padding 位在 softmax 前变成 `-inf`
- 裁剪：优先保留最近上下文、系统指令、关键实体、结构化状态
- 维度变换：本质上就是线性层把输入从 `d_model` 映到每个 head 的 `dk/dv`

这题还可以顺手连回 LoRA：

| LoRA target | 在注意力里改的是什么 |
| --- | --- |
| `q_proj` | 改“看什么” |
| `k_proj` | 改“怎么被匹配到” |
| `v_proj` | 改“取回什么内容” |
| `o_proj` | 改“怎么汇总回主干” |

## 19. 你提到基于 LLaMA-Factory 去做 LoRA 微调，这个是什么？

### 【面试当场如何回答】

LLaMA-Factory 更准确地说是一个**面向大模型微调和推理评测的训练工具链**，不是模型本身。  
在这个项目里，我主要用它做了两件事：一是跑 `router_lora` 和 `spot_lora` 的训练；二是把训练产物、日志、checkpoint 和离线评测串起来。  
它解决的是训练流程问题，不替你解决数据质量和评测可信度，所以真正让 Router LoRA 有明显提升的，还是任务定义、数据形式和 benchmark 设计。

### 【详细解释】

你可以这样定义：

**它不是 Qwen，也不是 LLaMA，而是帮助我们做 SFT、LoRA、模板适配、checkpoint 管理和本地推理评测的工具系统。**

当前项目里主要用途：

| 用途 | 当前体现 |
| --- | --- |
| LoRA 训练 | `router_lora_2026-03-12`、`spot_lora_2026-03-12` |
| 训练配置管理 | `training_args.yaml`、`adapter_config.json`、`trainer_state.json` |
| 模板适配 | `template = qwen3_5_nothink` |
| 本地评测 | 训练后结合脚本做 benchmark 对比 |

它为什么适合当前项目：

- 当前不是做超大规模预训练
- 主要是做角色化 SFT / LoRA
- 需要本地离线跑 benchmark
- 需要方便看 checkpoint、loss、配置

一定要补一句：

**LLaMA-Factory 只是把训练流程跑顺，不会自动替你解决数据是否高质量、指标是否可信、checkpoint 是否该选中间这一类工程判断。**

## 主要对比表速记

### 1. Prompt / LoRA / RAG / 工具

| 能力 | 解决什么问题 |
| --- | --- |
| Prompt | 把规则告诉模型 |
| LoRA / SFT | 把角色和流程更稳定地学进参数 |
| RAG | 补外部知识 |
| 工具 | 拿实时事实 |

### 2. MCP / Function Calling

| 概念 | 一句话理解 |
| --- | --- |
| MCP | 把工具接进系统 |
| Function Calling | 让模型选对工具和参数 |

### 3. 多 Agent / 单 Agent

| 维度 | 多 Agent | 单 Agent |
| --- | --- | --- |
| 边界 | 清晰 | 容易混 |
| 权限控制 | 容易隔离 | 容易越权 |
| 问题定位 | 好拆 | 容易混成一团 |
| 扩展 | 易加专家角色 | 容易越改越重 |
| 时延 | 更高 | 更低 |

### 4. OpenAgents / LangGraph

| 维度 | OpenAgents | LangGraph |
| --- | --- | --- |
| 更适合 | 消息驱动多 Agent | 强确定性状态图 |
| 当前项目匹配度 | 高 | 也能做，但不是当前首选 |

## 最后提醒

1. `Router LoRA` 是你最硬的证据，优先讲。
2. `Spot LoRA` 不要硬说“全面提升”，要说“风格增强，strict factual 指标下降，说明训练目标和评测目标没完全对齐”。
3. `RAG` 不要吹成“标准 BM25 + rerank”，要老实说“FAISS + BM25-like + bonus 的 Hybrid RAG”。
4. 记忆模块要主动补一句：**当前还是单文件全局记忆，不是按用户/线程隔离。**
5. 面试里最加分的不是全答对，而是你能主动区分：
   - 什么是当前真实实现
   - 什么是已经验证有效
   - 什么是当前识别出的技术债和下一步改进
