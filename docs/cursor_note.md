## 一、这次实际完成的工作

围绕项目里的 `RAG` 检索能力，我这次主要做了四件事：

### 1. 重构了检索评测链路

原来的评测方式主要是：

- 调 `search_knowledge()` 返回一段格式化文本
- 再用字符串查找 `expected_id`

这个方式的问题是：

- 不是真正的 `Top-K` 检索评测
- `Top-2` / `Top-5` 的定义不严格
- 很难做后续的分类分析、宽松评测和图文报告

因此我做了这一步重构：

- 在 `tools/spot_tools.py` 中新增结构化检索接口 `retrieve_knowledge()`
- 保留原有 `search_knowledge()` 文本接口不动，避免影响 MCP / agent 调用
- 让评测直接基于结构化 `Top-K` 结果做统计

相关文件：

- `tools/spot_tools.py`
- `tools/eval_rag.py`

### 2. 把 RAG 评测升级成了真正的离线 benchmark

评测脚本现在支持：

- `Hit@1 / Hit@3 / Hit@5`
- `MRR`
- 分类统计
- 未命中样例分析
- `results.json` 机器可读结果导出
- `report.md + PNG` 图文报告输出

这意味着现在不是“看看命中了没有”，而是已经有一套像样的离线评测基建。

最新评测主脚本：

- `tools/eval_rag.py`

### 3. 扩展了评测集规模

一开始只有很小的一批手工 case，单独看没有太强统计意义。

我把 `tools/generate_eval_dataset.py` 改成了一个纯离线、可复现的扩展生成器：

- 保留已有手工基线 case
- 基于 `storage/doecment.json` 做分层抽样
- 自动构造三类 query：
  - `精确景点`
  - `城市+景点`
  - `城市级多答案`

最终生成：

- `tools/expanded_test_cases.json`

目前扩展后的测试集规模为：

- 总计 `198` 条
- `手写`：12
- `Kaggle`：15
- `城市级`：38
- `精确景点`：63
- `城市+景点`：70

相关文件：

- `tools/generate_eval_dataset.py`
- `tools/expanded_test_cases.json`

### 4. 增加了“宽松实体级评测”

这是本次非常关键的一步。

当前知识库里同一个景点经常有多来源重复条目，例如：

- `kaggle_*`
- `spot_*`

如果严格按 `doc id` 算命中，那么：

- 用户问“故宫”
- 检索命中 `spot_beijing_001`
- 但标注答案是 `kaggle_北京_001`

这种情况会被算成 `miss`，但从真实用户体验上其实不合理。

所以我在 `tools/eval_rag.py` 里新增了第二套口径：

- `严格 Doc-ID 评测`
- `宽松实体级评测`

宽松评测规则：

- 若命中的是“同城市 + 同景点实体”的跨来源条目，也算命中

这样报告里就能把两套指标并列展示，后续写简历或答辩时能说清楚“口径”。

## 三、这组结果说明了什么

结论不能简单说“RAG 好”或者“RAG 坏”，而应该拆开看：

### 1. RAG 对私域内容是有价值的

因为手写私域攻略命中率明显高，说明：

- 这套本地知识库不是摆设
- 对项目里最有特色、最能体现“私域补充价值”的内容，RAG 确实能用

### 2. 对全国景点通用精确召回还不够稳

从 `精确景点` 和 `城市+景点` 的结果看：

- 命中率还偏低
- 说明真正的瓶颈仍在“精确实体召回”

### 3. 当前问题不只是排序，更是召回

因为整体上：

- `Hit@1` 和 `Hit@5` 差距不大

这意味着很多 case 不是“排得靠后”，而是“前 5 里都没召回到正确候选”。

所以当前最需要优化的是：

- 召回质量
- 实体归一
- 同景点多来源重复条目处理

## 四、这次做的一个小调优

我还对 `tools/spot_tools.py` 做了一个很保守的小修改：

- 把手写攻略加权从 `50` 调到 `35`

原因：

- 原来手写攻略有明显特权分
- 容易在一些场景里挤掉公域景点
- 但又不能一刀砍没，否则私域优势会掉

我先做了多档测试，确认：

- 从 `50` 降到 `30`
- 当前 `198` 条评测集结果都没有变化

所以最终选了 `35`，作为“适度去 bias”的安全值。

相关文件：

- `tools/spot_tools.py`

## 五、如果后面继续做，最值得的方向

优先级从高到低建议如下：

### 1. 做景点实体归一

把同一景点的：

- `spot_*`
- `kaggle_*`

绑定到统一的 `canonical id`

这样会直接改善：

- 检索
- 宽松评测
- 结果解释性

### 2. 强化精确景点召回

重点优化：

- 景点别名匹配
- 城市 / 区县约束
- exact match boost
- 中文分词后的 BM25

### 3. 扩大候选集后做 rerank

当前如果前 5 都没召回到正确候选，单纯靠现有排序已经不够。

后面可以考虑：

- 先召回更大候选集
- 再做二阶段 rerank

### 4. 持续扩 benchmark

现在 `198` 条已经比最早的小样本强很多，但仍然属于开发期 benchmark。

后面建议扩到：

- `300 ~ 500` 条

重点补：

- 口语化问法
- 约束型问法
- 同名景点
- 多来源同景点
- 易混淆城市 / 区县
