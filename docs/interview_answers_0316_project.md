## 1. 怎么解决幻觉，输出慢怎么解决？

如果面试官问我这个问题，我会先把“幻觉”拆成两类，因为解决办法不一样。

### 1.1 幻觉分两类

第一类叫**事实型幻觉**，比如天气、路线、酒店、景点地址这种本来应该查工具或知识库的数据，模型却自己编出来了。

第二类叫**行为型幻觉**，比如 Router 本来只该做意图分发，结果它越权自己查天气、自己答路线；或者专家 Agent 应该调用 `send_channel_message` 发消息，结果直接吐纯文本。这类问题本质上不是“知识错了”，而是“动作错了”。

### 1.2 这个项目里我是怎么压幻觉的

我实际用了四层约束。

#### 第一层：把强事实问题改成“必须走工具”

这个项目里最核心的思路不是让模型“尽量别编”，而是**把容易编造的任务从自由生成改成受约束的函数调用**。

- 天气问题必须走 `get_weather`
- 路线问题必须走 `get_driving_route`
- 酒店问题必须走 `search_hotels`
- 景点泛推荐优先走高德 POI 的 `search_spots`
- 具体攻略、避坑、玩法再补本地 RAG

原理上可以写成：

`y* = argmax_y p_theta(y | x)`

如果输入 `x` 里没有外部事实约束，模型就只能依赖参数记忆去“补全最像答案的东西”，这时就容易出现幻觉。  
而工具调用相当于先做：

`z = Tool(x)`

再做：

`y* = argmax_y p_theta(y | x, z)`

这里的 `z` 是外部真实结果，模型生成时不再只靠参数记忆，而是有了 grounding。

#### 第二层：职责拆分，限制 Agent 的权限边界

这个项目不是一个大模型包打天下，而是把权限拆开了：

- `travel_router` 只能做路由、指代消解、记忆读写
- `weather_agent` 只管天气
- `spot_agent` 只管景点/攻略
- `plan_agent` 只管路线、酒店、行程

例如 `travel_router.yaml` 里明确禁止 Router 直接调用天气、景点、酒店、路线等业务工具，只允许它通过 `send_direct_message` 转给专家 Agent。  
这能显著降低“路由器自己越权回答”的行为型幻觉。

#### 第三层：景点场景不用单一知识源，而是混合检索

景点场景最容易“半真半假”，因为它既包含 POI 类结构化信息，也包含攻略类非结构化知识。

所以我现在做的是 hybrid：

- 结构化 POI：高德地图结果，适合“这个城市有什么景点”
- 本地 RAG：适合“某个景点怎么玩、避坑、拍照机位、美食建议”

也就是说，泛推荐先拿真实 POI 保底，深度玩法再从本地知识库补充。这样比只靠模型回忆景点介绍稳得多。

#### 第四层：用评测和 badcase 反推系统漏洞

我不是只看主观回答“像不像”，而是做了专项评测：

- Router 的 tool-calling 评测
- Spot 的回答质量评测
- RAG 的 Hit@K / MRR 检索评测
- badcase 文档沉淀

如果某个阶段出现“回复很自然，但动作不对”或者“看起来很像答案，但槽位不准”，我会把它归因到对应模块，而不是笼统地说“模型不太行”。

### 1.3 这个项目里输出慢的来源是什么

总延迟可以近似拆成：

`T_total ≈ T_prefill + T_decode + T_tool + T_network + T_retrieval + T_orchestration`

其中：

- `T_prefill`：长上下文进入模型的预填充时间
- `T_decode`：逐 token 生成时间
- `T_tool`：工具执行时间
- `T_network`：外部 API 网络时间
- `T_retrieval`：向量检索/加载时间
- `T_orchestration`：Router -> 专家 Agent -> 频道回复 的编排时间

这个项目里，慢主要不是一个原因，而是几类时间叠加：

1. 多 Agent 串行转发，本来就比单轮直接回答多一跳。
2. 天气工具链是“高德地理解析 + Open-Meteo”，存在网络耗时。
3. 景点深度查询可能会走本地向量检索和融合排序。
4. 如果上下文过长，模型 prefill 会变慢。

### 1.4 我会怎么优化输出速度

这个项目里比较实用的优化方向有这些：

1. **缩短不必要上下文**  
   Router 负责读结构化记忆，不把所有历史原文都塞进模型。因为注意力复杂度在标准自注意力下接近：
   `O(n^2)`  
   上下文越长，prefill 越慢。

2. **让简单任务走最短路径**  
   例如景点城市级泛推荐优先 `search_spots`，而不是无脑再加本地 RAG。2026-03-15 的 POI-first 修复，本质上既提高稳定性，也减少不必要检索。

3. **把事实查询和文案生成分开**  
   先查工具，再做较短回答，比让模型一边“想工具”、一边“组织长文”更稳。

4. **缓存与懒加载**  
   当前景点向量索引就是懒加载。再往前可以继续做 geocode 缓存、热门城市天气短时缓存、查询结果缓存。

5. **按任务微调而不是一味换更大模型**  
   Router LoRA 的意义之一就是让模型更快进入正确动作分布，减少反复试错和格式漂移。

### 1.5 我会怎么总结这一题

这题我不会回答成“靠提示词防幻觉”。更准确的回答应该是：

**这个项目解决幻觉的核心不是口头约束，而是把高风险事实任务改造成工具调用，把复杂任务拆给专家 Agent，再用 RAG/POI 和结构化记忆做 grounding，最后用评测闭环去校正。输出慢则从上下文长度、外部工具延迟、检索链路和多 Agent 编排四个方向优化。**

## 2. Agent 间数据怎么流动？结合项目举例：travel_router 如何把天气问题交给 weather_agent？

这题我会直接按当前项目的真实链路讲，不讲抽象空话。

### 2.1 当前项目不是“广播式多 Agent”，而是“Router + Direct Message 专家”

当前实际架构是：

- 用户先在公共频道发消息
- `travel_router` 监听公共频道消息
- Router 读记忆、做指代消解、识别意图
- Router 用 `send_direct_message` 把任务发给目标专家 Agent
- 专家 Agent 只监听来自 Router 的 `thread.direct_message.notification`
- 专家 Agent 调工具后，用 `send_channel_message(channel="general", text=...)` 回公共频道

所以它不是“所有 Agent 同时听见用户广播，然后抢答”，而是**Router 集中分发，专家定向处理**。

### 2.2 触发机制是怎么配的

按当前 YAML 配置：

- `travel_router` 处理：
  - `thread.channel_message.notification`
  - `thread.reply.notification`
- `weather_agent` / `spot_agent` / `plan_agent` 处理：
  - `thread.direct_message.notification`

而且每个 Agent 都有发送者过滤规则。

例如 Router 明确规定：

- 如果消息发送者是 `travel_router`、`weather_agent`、`spot_agent`、`plan_agent`
- 说明这是别的 Agent 发出来的消息
- 它必须 `finish`
- 不能把别的 Agent 回复再当成新用户需求重新处理

这一步很关键，不然系统会出现“Agent 听到别的 Agent 的输出后再次路由”的回声循环。

### 2.3 以天气问题为例，完整数据流如下

假设用户问：

`“故城县今天天气如何？”`

那么实际链路可以写成下面这样：

```text
用户 -> 公共频道 general
   -> travel_router 收到 channel_message
   -> travel_router 调用 get_context 读取记忆
   -> travel_router 判断为天气意图
   -> travel_router 调用 send_direct_message(target_agent_id="weather_agent", text="故城县今天天气如何？")
   -> travel_router 视情况调用 save_context(current_city="故城县")
   -> weather_agent 收到 direct_message
   -> weather_agent 调用 get_context 获取上下文
   -> weather_agent 抽取城市名，必要时用记忆补全
   -> weather_agent 调用 get_weather(city="故城县")
   -> weather_agent 调用 send_channel_message(channel="general", text="天气结果")
   -> 用户在公共频道看到结果
```

### 2.4 这里的数据到底“流动”了什么

这一题很多人容易只说“消息在流动”，但真正应该讲清楚的是**流动的数据类型**。

这个项目里至少有三类数据在流动：

#### 第一类：自然语言任务数据

比如：

- “故城县今天天气如何？”
- “从故城县到如皋市，今天出发，有三天时间，请帮我规划行程。”

这是 Router 转发给专家 Agent 的主要载体。

#### 第二类：结构化状态数据

记忆模块保存的是结构化字段，而不是整段聊天原文。当前 `storage/user_context.json` 里核心字段包括：

- `current_city`
- `origin_city`
- `destination_city`
- `current_spot`
- `travel_party`
- `preferences`
- `departure_time`
- `trip_days`
- `notes`
- `last_updated`

这类数据让系统能做指代消解和多轮补全。

#### 第三类：工具返回的外部事实数据

例如天气工具返回温度、体感、风速、湿度；路线工具返回距离和耗时；酒店工具返回真实 POI 列表；景点工具返回 POI 或 RAG 命中结果。

也就是说，这个系统不是“模型凭空想”，而是：

`用户语言输入 -> Router 转成正确动作 -> 工具产出外部事实 -> 专家 Agent 组织回复`

### 2.5 Router 的核心工作其实是三件事

我会把 Router 的职责概括成三步：

1. **读记忆**
2. **做指代消解和意图识别**
3. **把任务发给正确 Agent，并保存新状态**

这里尤其要强调“指代消解”，因为这正是多轮对话里最有价值的地方。

比如用户前面说：

`“我想从故城县去如皋市。”`

后面再说：

`“那边住哪里方便？”`

Router 不应该把“那边”原样转发，而应该结合记忆里的 `destination_city="如皋市"`，补全成：

`“如皋市住哪里方便？推荐几个酒店。”`

再交给 `plan_agent`。

### 2.6 当前项目里 Router 为什么不能自己直接查天气

因为 Router 的职责是“总控”，不是“业务执行器”。

如果 Router 又路由、又查天气、又查路线、又查酒店，会有三个问题：

1. Prompt 会越来越大，行为边界会变模糊。
2. 评测会变难。你很难判断错在“意图分发”还是“业务生成”。
3. 一个角色失败会污染整个系统，无法局部替换。

所以这个项目是显式把业务工具权限下放给专家 Agent，把 Router 限制成轻中台。

### 2.7 当前实现里还有什么限制

这题如果只讲优点不讲限制，会显得不真实。我会补一句当前局限：

1. 记忆文件目前是单个 `storage/user_context.json`，还不是按用户/线程隔离的。
2. 多 Agent 链路天然比单 Agent 多一跳，时延会增加。
3. 专家 Agent 之间没有复杂协商机制，现在主要是 Router 单点分发。

### 2.8 这一题可以怎么收束

我会总结成一句：

**这个项目的 Agent 数据流不是简单“消息传来传去”，而是公共频道事件、direct message、结构化记忆、工具返回事实三种数据共同流动。Router 负责读状态、消解指代、定向分发；专家 Agent 负责调工具和最终回复。**

## 3. LoRA 微调和改提示词有什么区别？你怎么评估？你到底有没有真正做 LoRA？超参数怎么定？为什么有的 checkpoint 不是 final？

这题我会分成六部分回答：**原理差异、我实际做了什么、怎么评估、超参数怎么选、LoRA 和全参微调怎么比较、为什么选中间 checkpoint。**

### 3.1 Prompt 优化和 LoRA 微调，本质上改的是两件不同的事

#### Prompt 优化

Prompt 优化本质上是**不改模型参数，只改输入条件**。  
它的作用是把任务规则、角色边界、输出格式、few-shot 示例告诉模型，让模型在现有能力范围内更稳定地表现。

#### LoRA 微调

LoRA 是**在冻结原模型大部分参数的前提下，只训练一个低秩增量**。  
常见写法是：

`W' = W_0 + Delta W`

其中：

`Delta W = (alpha / r) * B A`

- `W_0`：原始权重
- `A ∈ R^(r x d_in)`
- `B ∈ R^(d_out x r)`
- `r`：低秩 rank
- `alpha`：缩放系数

它改的是模型参数本身，所以改变的不只是“这轮怎么说”，而是模型对某类任务的整体行为分布。

#### 两者的真正区别

一句话讲清楚就是：

- Prompt 调的是“条件”
- LoRA 调的是“参数”

所以在这个项目里，Prompt 负责规定流程，LoRA 负责让模型更稳定地学会这个流程。

### 3.2 这个项目里我不是只学了原理，而是真的做了两套 LoRA

当前仓库里至少有两套明确的 LoRA 产物：

1. `models/router_lora_2026-03-12`
2. `models/spot_lora_2026-03-12`

也就是说，我不是只会背 LoRA 原理，而是实际做了：

- 数据准备
- LLaMA-Factory 训练
- checkpoint 选择
- 离线评测
- badcase 分析

### 3.3 我做 LoRA 的目标不是“泛泛提升模型能力”，而是提升特定任务指标

#### Router LoRA 的目标

Router LoRA 的目标非常明确：  
让 Router 学会**正确读记忆、正确分发、少犯工具调用错误、少漏关键参数**。

所以我看的是 tool-calling 类指标，而不是只看语言自然度。

按仓库里原始评测报告：

基础模型（`0314eval/report (2).md`）：

- tool-call emission：`97.62%`
- tool-name accuracy：`51.19%`
- arg exact：`19.57%`
- turn exact：`48.81%`
- decision tool-name acc：`19.57%`
- sample full-chain success：`20.59%`

Router LoRA（`0314eval/report_new.md`）：

- tool-call emission：`100%`
- tool-name accuracy：`100%`
- arg exact：`89.13%`
- turn exact：`94.05%`
- decision tool-name acc：`100%`
- sample full-chain success：`85.29%`

这个提升是非常实打实的，不是主观感觉。

#### Spot LoRA 的目标

Spot LoRA 的目标是让景点回答更像旅游助手、更会组织内容。  
但它的结果是**风格和可读性变好了，严格槽位命中反而有下降**。这说明 Spot 这个任务不是“随便 SFT 一下就全面变好”，后面我在第 5 题专门展开。

### 3.4 我怎么评估 LoRA 效果

#### Router 评估

Router 评估是动作导向的，所以看：

- 是否发起工具调用
- 工具名是否对
- 参数是否完全正确
- 整轮动作链是否正确
- 最终完整样本是否成功

这是因为 Router 的核心不是写长文，而是**做对动作**。

#### Spot 评估

Spot 评估里用了：

- strict success
- fact coverage
- ROUGE-L F1

这里可以补一个公式。  
ROUGE-L F1 本质上还是 Precision 和 Recall 的调和平均：

`F1 = 2PR / (P + R)`

只是它的 P、R 是基于最长公共子序列的文本重叠来算。

#### 训练目标本身

SFT 训练时优化的是交叉熵损失：

`L_SFT = - Σ_{t=1}^T log p_theta(y_t | x, y_<t)`

这说明训练时直接优化的是“参考答案 token 概率”，不是最终业务指标。  
所以真正的工程闭环一定是：

`训练损失 + 下游任务评测 + badcase 复盘`

三者一起看。

### 3.6 这些超参数不是拍脑袋定的

我会这样解释：

1. `r = 8`：先选一个中等偏小、工程上常见、显存可控的 rank。
2. `alpha = 16`：让缩放比例 `alpha / r = 2`，更新幅度不会太小。
3. `dropout = 0.05`：给一点正则，但不要大到把有效学习信号冲掉。
4. Router 学习率比 Spot 高一点，因为 Router 任务更偏动作模式学习；Spot 更偏自然语言表达和内容组织，学习率保守一些。
5. `cosine scheduler + warmup`：让训练前期更稳，后期逐步收敛。

### 3.7 全参微调和 LoRA 的区别、优劣、选择

#### 全参微调

优点：

- 理论上容量最大
- 能更彻底改变模型行为

缺点：

- 显存和存储成本高
- 训练慢
- 更容易把通用能力一起带偏
- 多任务迭代时管理成本大

#### LoRA

优点：

- 显存友好
- 训练快
- adapter 可插拔
- 适合按角色分别训练，比如 Router 一个 adapter、Spot 一个 adapter

缺点：

- 容量有限
- 某些复杂行为可能不如全参彻底
- 如果数据目标和评测目标不一致，LoRA 也会学偏

#### 这个项目为什么优先选 LoRA

因为这个项目要解决的是**特定角色、特定行为模式**：

- Router：学会稳定路由和工具调用
- Spot：学会景点回答风格和格式

这类任务非常适合 LoRA，而没有必要一上来就做全参微调。

### 3.8 为什么有的 checkpoint 不是 final，而是中间 checkpoint

这是工程上非常正常的现象。

原因是：**最低训练/验证损失，不一定等于最佳下游业务指标。**

比如 Router 这类任务，真正想要的是：

- 工具名对不对
- 参数准不准
- 整轮是否执行对

这和单纯 token-level loss 不是完全同一个目标。

当前仓库里的 Router 下游 benchmark 实际采用的是 `checkpoint-60`。  
也就是说，虽然训练还能继续、甚至 final loss 还可能更低，但在“真实工具调用行为”这个目标上，中间 checkpoint 反而最好。

这是因为模型训练后期可能会继续贴合训练文本分布，却不一定继续提升离散动作边界。

### 3.9 损失分析我会怎么讲

Router 的 `trainer_state.json` 可以看到 eval loss 从大约：

- step 20：`0.1806`
- step 40：`0.0657`
- step 60：`0.03885`

最终 eval 结果约为：

- `0.03775`

这说明训练是明显收敛的。  
但是我不会只拿 loss 讲故事，我会补一句：

**Router 任务更重要的是行为正确率，所以最终 checkpoint 的取舍，要以下游 tool-calling benchmark 为主，而不是单看谁 loss 最低。**

### 3.10 这一题最后怎么收束

我会总结成：

**Prompt 优化解决的是“怎么把规则告诉模型”，LoRA 解决的是“让模型把这个规则真正学进参数里”。这个项目里我确实做了 Router 和 Spot 两套 LoRA，并且用下游评测而不是主观感觉来选 checkpoint。Router LoRA 提升非常明显；Spot LoRA 则暴露了“语言风格优化”和“严格事实命中”之间的张力。**

## 4. 为什么 rank 常设成 8、16？alpha 为什么常是 rank 的两倍？target module 有什么作用？加在 q/k/v/o 上分别意味着什么？

这题本质上是在问 LoRA 的参数化机制，不是背经验值。

### 4.1 rank 到底控制什么

LoRA 的核心假设是：  
很多任务所需的权重更新 `Delta W` 不需要满秩，可以用低秩矩阵近似。

也就是：

`Delta W ≈ (alpha / r) * B A`

其中 `rank = r` 决定了这个增量更新所在子空间的维度。

如果把原始线性层看成：

`W ∈ R^(d_out x d_in)`

那么 LoRA 新增参数量大约是：

`Params_LoRA ≈ r * (d_in + d_out)`

可以看到，它和 `r` 近似线性相关。

所以 `r` 不是随便写个数字，它直接影响两件事：

1. **表达能力**：r 越大，可学习的更新子空间越大。
2. **资源消耗**：r 越大，参数量、显存占用、训练时间都上升。

### 4.2 为什么很多工程里常从 8、16 开始

不是因为 8、16 有神秘理论，而是因为它们是一个常见的工程折中点：

- 太小，比如 1、2、4，容量可能不够
- 太大，比如 64、128，对很多任务不一定值得
- 8、16 往往已经能覆盖很多“风格调整、工具调用模式、角色行为”的适配需求

这个项目里 Router 和 Spot 都用了 `r = 8`，原因就是：

1. 任务不是重新学世界知识，而是学角色化行为
2. 需要控制本地训练成本
3. 先从中低 rank 验证收益，再决定是否扩容

### 4.3 rank 对效果真的有影响吗

有影响，而且不只是影响显存。

可以把它理解成：  
你让模型能改动的“自由度”有多大。

- `r` 太小：可能学不会足够细的行为边界，表现成欠拟合
- `r` 适中：能学到任务模式，又不至于过重
- `r` 太大：虽然容量更强，但也更容易把训练集风格学过头，泛化未必更好

所以它和效果当然有关，只是不是“越大越好”。

### 4.4 alpha 的作用是什么

`alpha` 是 LoRA 增量的缩放系数。  
真正加到原权重上的量是：

`(alpha / r) * B A`

所以决定更新有效幅度的，不是单看 `alpha`，而是看：

`scale = alpha / r`

比如本项目里：

- `r = 8`
- `alpha = 16`

那么：

`scale = 16 / 8 = 2`

这意味着 LoRA 学到的低秩更新会按 2 倍缩放后叠加到原模型上。

### 4.5 为什么很多人喜欢设成 rank 的两倍

这不是硬性理论定律，而是社区里常见、相对稳妥的经验设定。

更准确地说，不应该把它讲成“原论文一倍，后来发现梯度太小，所以改两倍”。  
更严谨的表达是：

1. 原始 LoRA 论文里不同实验并不是只有一种固定比例。
2. 工程实践里大家发现如果 `alpha / r` 太小，增量影响可能偏弱。
3. 设成 `alpha = 2r`，也就是 `scale = 2`，通常能让训练更容易学出可见效果，而且一般比较稳。

所以“alpha 常设成 rank 两倍”更像是**经验上常用的缩放习惯**，不是必须遵守的物理定律。

### 4.6 target module 为什么重要

因为 LoRA 不是对整个模型所有参数都加低秩分支，而是只加在某些线性层上。  
加在哪里，决定你在改模型的哪一部分能力。

这个项目里配置是 `lora_target = all`，所以不只是注意力里的 `q/k/v/o`，还覆盖了更多线性层。这样做的考虑是：任务不仅有注意力重排，也有输出组织和 MLP 表达变换。

### 4.7 加在 q/k/v/o 上分别意味着什么

先写注意力的标准公式：

`Q = X W_Q`

`K = X W_K`

`V = X W_V`

`Attention(Q, K, V) = softmax(Q K^T / sqrt(d_k)) V`

输出再经过：

`O = Attention(...) W_O`

所以：

#### 加在 `q_proj`

影响的是 Query，也就是“当前 token 去关注什么”。  
更偏向改变**提问方式、关注点、对齐模式**。

#### 加在 `k_proj`

影响的是 Key，也就是“别的 token 以什么特征被匹配到”。  
更偏向改变**可匹配性和索引方式**。

#### 加在 `v_proj`

影响的是 Value，也就是“被取回的信息内容是什么表示”。  
更偏向改变**被聚合的信息内容形态**。

#### 加在 `o_proj`

影响的是多头注意力汇总后的输出投影。  
更偏向改变**多个 head 结果如何重新混合并传到后续层**。

### 4.8 为什么很多人至少会放 q、v，有些再加 k、o

因为从工程经验看：

- 只改 `q/v`，往往已经能带来明显收益，成本也较低
- 加上 `k/o`，适配能力更强，但参数量会继续增加
- 如果任务偏格式、风格、工具调用边界，`q/k/v/o` 一起改通常更稳

这个项目最终用了更宽的 target 范围，本质是希望模型在“角色行为 + 输出形式 + 领域表达”上都能一起适配，而不是只改一点点注意力行为。

### 4.9 这题我会怎么收尾

**rank 决定 LoRA 更新子空间的容量，alpha 决定更新幅度，target module 决定你到底改模型的哪一部分能力。8、16 这类值不是玄学，而是容量、泛化、显存三者之间的工程折中；alpha 设成 rank 的两倍也不是定律，而是常见且相对稳定的经验值。**

## 5. 为什么加了个东西，模型效果反而变差？Spot LoRA 为什么有指标下降？问题出在哪，怎么改？

这题我会非常诚实地答，因为它恰恰体现我不是只会报喜不报忧。

### 5.1 先说结论：Spot LoRA 不是“全面变差”，而是“指标发生了结构性变化”

按仓库里可复核的原始 CrossWOZ 对比结果：

基础模型（`0314eval/report (3).md`）：

- strict success：`98.00%`
- avg fact coverage：`99.00%`
- avg ROUGE-L F1：`0.0266`

Spot LoRA（`0314eval/report (4).md`）：

- strict success：`78.00%`
- avg fact coverage：`89.11%`
- avg ROUGE-L F1：`0.2736`

这说明什么？

说明它不是简单地“全都更差”，而是：

1. **严格槽位命中下降了**
2. **事实覆盖也下降了一些**
3. **文本表达和参考答案表面相似度明显提高了**

换句话说，Spot LoRA 让回答更像“会说话的旅游助手”，但不一定更像“严格对齐 benchmark 槽位的抽取器”。

### 5.2 为什么会出现这种现象

我会把原因拆成四层。

#### 第一层：训练目标和评测目标不完全一致

SFT 优化的是：

`L_SFT = - Σ log p_theta(y_t | x, y_<t)`

也就是 token 级条件概率。

但 Spot 评测里的 strict success 更接近“离散槽位全对才算对”。  
这两个目标不是严格等价的。

所以模型可能学会了：

- 回答更自然
- 句子更完整
- 更会组织旅游文案

但它未必学会了：

- 必须把景点名点出来
- 必须把游玩时长写成评测脚本能识别的格式
- 必须把门票、评分、地址这些槽位按预期覆盖全

#### 第二层：数据分布更偏“攻略表达”，不是纯“槽位抽取”

当前 Spot LoRA 的训练数据文件是 `data/lora_ready/02_spot_cultour_clean.json`，共有 `8802` 条。  
从文件命名、训练任务和相关脚本看，它明显更偏向景点/文旅问答和攻略表达，而不是纯结构化槽位生成。

这就意味着模型会更倾向学到：

- 旅游文案风格
- 推荐口吻
- 叙述性说明

而不是只学：

- 名称
- 地址
- 门票
- 游玩时间
- 评分

如果评测更偏槽位精确匹配，风格增强反而可能拖 strict 指标。

#### 第三层：模型责任边界不够清晰

Spot 这个任务里，系统同时希望模型做到三件事：

1. 读懂检索上下文
2. 抽准事实槽位
3. 说得像一个自然的旅游助手

这三件事其实不完全同向。  
尤其当数据规模没有大到足够覆盖各种“事实抽取格式”时，模型很容易把精力花在更容易优化的部分，也就是语言组织和风格迁移。

#### 第四层：这更像“风格过拟合 + 槽位约束不足”，而不是单一的过拟合或欠拟合

很多人会问，到底是过拟合还是欠拟合？  
我会说：**不能粗暴只选一个词。**

更准确的说法是：

- 对“旅游回答风格”这件事，模型学得比较重，甚至有点过头
- 对“严格槽位边界”这件事，模型约束又不够强

所以它表现成一种混合现象：

- 对风格分布来说，像过拟合
- 对结构化精度来说，又像没学到位

### 5.3 训练日志能说明什么

Spot 的训练日志显示 eval loss 大体是持续下降的，大致从：

- step 50：`1.9469`
- 逐步降到
- step 500 左右：`1.8405`
- 最终约：`1.8407`

这说明训练过程本身没有明显发散。  
也就是说，问题不在于“训练炸了”，而更可能在于：

1. 训练目标和业务目标错位
2. 数据分布和评测需求错位
3. 输出格式约束不够

### 5.4 为什么它“没别人的好”

如果面试官追问“为什么没别人的好”，我不会泛泛说“数据不够多”，而会具体说这几个点：

1. **任务拆分不彻底**  
   当前 Spot 既要像 RAG 问答器，又要像旅游推荐文案助手。  
   最好把“事实抽取”与“风格生成”拆成两阶段。

2. **训练样本对 strict 评测不够对齐**  
   如果 benchmark 关心固定槽位，就应该专门补“短答案、强约束、字段完整”的样本。

3. **缺少结构化中间表示**  
   现在基本还是直接从上下文到自然语言答案。  
   更稳的做法是先生成结构化事实，再渲染成自然语言。

4. **RAG/工具结果与生成风格之间缺少硬绑定**  
   模型会“理解后转述”，一转述就容易把评测脚本依赖的原始槽位格式改掉。

### 5.5 改进思路我会怎么提

这部分一定要讲得像真能落地，而不是空泛说“继续调参”。

#### 方案一：拆成“事实层”和“表述层”

先让模型输出一个受 schema 约束的中间结构，例如：

```json
{
  "spot_name": "...",
  "address": "...",
  "duration": "...",
  "ticket": "...",
  "rating": "..."
}
```

再把这份结构化结果渲染成自然语言。  
这样就把“事实对不对”和“话说得好不好”拆开了。

#### 方案二：补专门的 strict-format 数据

针对 badcase 最多的槽位补数据，尤其是：

- 游玩时间
- 门票
- 景点名称显式化
- 预算/时长类约束

也就是不要只补“旅游攻略文案”，要补“评测真正关心的格式”。

#### 方案三：训练时引入更强的字段覆盖约束

即使还是用 SFT，也可以通过数据模板把输出格式拉齐，例如明确要求：

- 必须出现景点名
- 必须出现门票信息
- 没有信息时明确写“未查到”，而不是省略

#### 方案四：分开做两个 Adapter

一个 Adapter 偏 facts，一个 Adapter 偏 style。  
或者更简单地说：

- Router/结构化任务用一个 adapter
- Spot 风格化问答用一个 adapter

甚至未来 Spot 里也可以细拆：

- “景点事实问答 LoRA”
- “攻略文案增强 LoRA”

### 5.6 这题最后该怎么总结

**Spot LoRA 的问题不在于“训练失败”，而在于训练目标、数据分布和评测指标三者不完全一致。它把回答风格学上去了，但严格槽位命中下降了。真正的改进方向不是盲目继续训，而是把事实抽取和自然语言表达解耦，并补更贴近评测目标的训练样本。**

## 6. RAG 你怎么实现的？怎么排查问题？为什么改了好几版？chunk 怎么切？用没用 rerank？怎么评估？

这题我会按“当前真实实现 -> 原理 -> 排查方法 -> 改版原因 -> 指标”来答。

### 6.1 先说一句最关键的话：当前项目里的景点 RAG，不是“纯本地向量库”，而是一个 hybrid 系统

当前真实实现里，景点相关能力实际上来自三部分：

1. **高德 POI**：结构化地点召回
2. **本地向量检索**：语义相似召回
3. **本地词项匹配分数**：一个 BM25-like 词法补分

所以如果让我一句话定义，我会说：

**这个项目当前更准确的说法是“高德 POI + FAISS 向量检索 + BM25-like 词法打分”的 hybrid 检索系统。**

注意，我会明确说是 **BM25-like**，而不是吹成标准 BM25，因为当前代码里的 `_bm25_score()` 是启发式词项打分，不是严格按标准 BM25 公式实现的。

### 6.2 本地 RAG 数据库是怎么搭的

当前知识库主文件是 `data/spots_knowledge.json`，目前共有 `15723` 条记录。

已知来源包括：

- Kaggle / 去哪儿网：`14581`
- CrossWOZ 景点库：`465`
- 独家手写攻略：`347`
- China312 地理数据集：`312`
- 香港旅游指南 QA：`18`

构建流程在 `tools/build_spot_vectors.py` 里，核心步骤是：

1. 读取结构化景点记录
2. 把 `city + district + spot_name + tags + content` 拼成检索文本
3. 用 `BAAI/bge-small-zh-v1.5` 生成 embedding
4. 做向量归一化
5. 存入 FAISS `IndexFlatIP`
6. 同时保存文档映射到 `storage/doecment.json`

因为做了归一化，内积检索就等价于余弦相似度检索：

`cos(x, y) = (x · y) / (||x|| ||y||)`

归一化后：

`||x|| = ||y|| = 1`

所以：

`x · y = cos(x, y)`

### 6.3 embedding 的意义是什么？和 one-hot 相比有什么本质区别？

这一段面试很容易问。

#### one-hot

one-hot 的特点是：

- 维度极高
- 极稀疏
- 不表达语义相似性

比如“景点推荐”和“旅游推荐”，在 one-hot 里可能几乎完全不相似。

#### embedding

embedding 是把文本映射到低维连续向量空间：

`e = f(text), e ∈ R^d`

它的意义是：  
语义相近的文本，在向量空间里也尽量靠近。

所以像：

- “衡水湖怎么玩”
- “衡水湖游玩攻略”
- “去衡水湖要注意什么”

这类问题即便表面词不完全一样，也能在向量空间里聚在一起。

这就是 embedding 相比 one-hot 的根本优势：  
**它能表达语义相似，而不是只表达字面是否相同。**

### 6.4 当前项目里的 chunk 是怎么切的

如果要按真实情况回答，我不会说“做了很高级的自适应 chunk”，因为当前实现不是那样。

当前更接近：

- **一条结构化景点/攻略记录作为一个检索单元**
- 不是长文档按 token 窗口切块
- 也不是根据段落语义做动态切分

也就是说，当前粒度基本是**record-level 固定粒度切分**，不是复杂的 adaptive chunking。

这样做的优点是：

- 实现简单
- 景点名、城市名、标签这类结构信息保留完整

缺点是：

- 如果某条 `content` 很长，可能信息密度不均
- 细粒度召回和局部精确定位能力有限

### 6.5 当前融合检索是怎么做的

按当前代码逻辑，可以近似写成：

`score(d, q) = 5.0 * dense_score(d, q) + bm25_like(d, q) + handwritten_bonus(d, q)`

其中：

- `dense_score(d, q)`：向量相似度
- `bm25_like(d, q)`：启发式词法匹配得分
- `handwritten_bonus(d, q)`：如果命中文档来源是“独家手写攻略”且有词项匹配，则额外加 `35.0`

也就是说，目前不是学术论文里那种特别标准化的 dense + sparse rerank pipeline，而是一个很工程化的混合打分器。

### 6.6 用没用 rerank

如果按当前仓库真实情况回答：  
**没有用独立 cross-encoder reranker。**

当前只有：

- FAISS 召回
- BM25-like 补分
- 手写攻略 bonus

所以如果面试官问“有没有 rerank”，我会说：

**严格意义上没有上单独的神经重排序模型，目前更像是融合打分，而不是单独的 rerank stage。**

### 6.7 Hybrid RAG 具体混合了什么

这题我会分成两层回答。

#### 第一层：本地检索内部的 hybrid

本地检索内部混合了：

- dense retrieval（embedding + FAISS）
- sparse / lexical retrieval（BM25-like）

#### 第二层：系统级 hybrid

系统整体还混合了：

- 远程结构化 API（高德 POI）
- 本地非结构化 RAG

所以这个项目里“hybrid”既指**密集检索 + 词法检索的混合**，也指**远程 POI + 本地知识库的混合**。

### 6.8 远程 API 和本地 RAG 的优先级怎么定

这是我 2026-03-15 之后专门修过的一点，核心思想是：**按意图决定优先级，而不是永远固定顺序。**

#### 城市级/区县级泛推荐

例如：

- “如皋市有什么好玩的”
- “故城县景点推荐”

这种问题现在优先走 **POI-first**，也就是先 `search_spots`。  
原因很简单：这种问题首先要解决的是“哪里有点位”，高德 POI 更稳。

#### 具体景点深度问题

例如：

- “衡水湖怎么玩”
- “西湖避坑有什么”
- “某个景点附近拍照机位”

这种问题本地 RAG 更有价值，因为 POI 只告诉你它“在哪”，不告诉你“怎么玩”。

#### POI 不足时补 RAG

如果高德返回结果太弱，或者用户明确要“攻略、避坑、玩法、美食、路线”，再补本地 RAG。

### 6.9 我为什么改了好几版 RAG

因为实际 badcase 不是一个点的问题，而是多点叠加。

我目前看到的主要改动动机有这几类：

1. **城市/地区解析错误**  
   以前把整句自然语言直接塞给高德 `city` 参数，会导致解析很脏。  
   后来加了 `_extract_location_hint` 去抽更可靠的地名 hint。

2. **城市级泛推荐误用本地 RAG**  
   会导致召回一堆“攻略文本”，却不一定先拿到真实可去的景点 POI。  
   所以后来改成了 POI-first。

3. **知识库覆盖不足**  
   后来又补了 CrossWOZ、香港旅游指南等数据源。

4. **需要抬高手写攻略权重**  
   因为“独家手写攻略”虽然数量不大，但质量高、区分度强，所以在融合里给了 bonus。

### 6.10 RAG 一般怎么排查问题

我会把排查拆成 5 层，层层定位。

#### 第一层：Query 有没有被错误改写

先看用户问题进入检索前有没有被错误解析。  
例如城市提取错了，后面全都白搭。

#### 第二层：召回有没有命中

看目标文档是否出现在 topK。  
如果 topK 根本没进，问题是召回。

#### 第三层：排序有没有把好文档压下去

如果目标文档进了 topK 但 rank 很低，问题在融合或排序。

#### 第四层：生成阶段有没有“看见了也没说对”

如果检索结果其实已经给了正确事实，但最终答案没用上，问题在生成。

#### 第五层：数据源本身够不够

如果知识库压根没有，或者数据源写得很脏，那不是检索算法能救的。

### 6.11 用什么指标衡量 RAG

当前仓库里有明确的检索指标：

#### Hit@K

`Hit@K = (1 / N) * Σ 1[rank_i <= K]`

表示正确文档是否出现在前 K 个结果里。

#### MRR

`MRR = (1 / N) * Σ (1 / rank_i)`

表示正确答案排得越靠前越好。

仓库里的 `reports/rag_eval_20260313_153914/report.md` 给出的结果基于当时 `15240` 条文档版本，主要数值是：

严格 Doc-ID：

- Hit@1：`35.9%`
- Hit@3：`36.9%`
- Hit@5：`36.9%`
- MRR：`0.3636`

宽松实体匹配：

- Hit@1：`42.9%`
- Hit@3：`43.9%`
- Hit@5：`43.9%`
- MRR：`0.4343`

还有一些分桶结果很有意思：

- 手写攻略 Hit@1：`91.7%`
- 城市级 loose Hit@1：`86.8%`
- 精确景点 strict Hit@1：`23.8%`

这说明：

1. 手写高质量数据很强
2. 城市级泛问题比较容易
3. 精确景点级别的 strict 检索仍然是短板

### 6.12 Hallucination 有没有单独指标

如果严谨回答：  
**目前仓库里没有一个完全独立、标准化的 hallucination rate 总表。**

当前更像是通过三种方式间接看：

1. RAG 的 Hit@K / MRR
2. Spot 回答评测的 strict success / fact coverage
3. 人工 badcase 复盘

所以我不会吹成“已经有完整幻觉率指标体系”，而会说：  
**当前已有检索指标和回答指标，但独立的 hallucination benchmark 还可以继续补。**

### 6.13 这题最后怎么总结

**这个项目的 RAG 不是单一向量库，而是高德 POI + 本地 FAISS + BM25-like 的 hybrid 检索系统。当前 chunk 以记录级固定粒度为主，没有独立 cross-encoder rerank。排查时我会按 query 解析、召回、排序、生成、数据源五层走。现有评测表明城市级问题较稳，但精确景点 strict 检索仍有明显优化空间。**

## 7. 如果 JSON 格式输出错了怎么办？

这题我不会只答“让模型重新生成一次”，那太浅了。

### 7.1 先说原则：尽量别把高风险动作建立在自由 JSON 上

这个项目当前的主线其实已经在做这件事了：  
**优先使用 Function Calling / MCP 工具调用，而不是让模型手写一大段 JSON。**

原因很简单，自由 JSON 常见错误包括：

- 少逗号
- 引号不匹配
- 键名漂移
- 字段漏填
- 中文描述混入结构体

而一旦你后面还要靠这个 JSON 驱动系统动作，风险就很大。

### 7.2 如果必须输出 JSON，我会做四层防护

#### 第一层：先定义 schema

不要只说“输出 JSON”，而要说清楚：

- 必填字段
- 字段类型
- 枚举范围
- 是否允许为空

例如：

```json
{
  "type": "object",
  "properties": {
    "city": { "type": "string" },
    "trip_days": { "type": "string" }
  },
  "required": ["city"]
}
```

#### 第二层：解析和校验分开

应该先：

1. `json.loads`
2. schema validate

不能解析成功就直接认为业务可用。

#### 第三层：失败时做 repair

当前仓库里就有一个 `tools/fix_json.py`，本质上是对脏 JSON 的离线修补脚本。  
虽然它不是一个通用完美方案，但已经体现了这个思路：

- 先尝试解析
- 不行就修常见格式错误
- 修完再解析

#### 第四层：repair 后仍失败，就回退或重试

如果 JSON 仍不合法，就不要硬吃。  
应该：

- 让模型按 schema 重试
- 或者回退为人工澄清
- 或者直接拒绝执行高风险动作

### 7.3 这一题和当前项目怎么结合

当前项目里我会强调：

1. **在线链路优先走 Function Calling / MCP**  
   也就是让模型输出结构化工具调用，而不是自己拼 JSON。

2. **离线数据清洗阶段允许使用 repair 脚本**  
   例如清洗训练数据或知识库中脏格式样本时，`fix_json.py` 这种工具是有价值的。

3. **如果要把 JSON 用于生产动作，必须做 schema 校验**

### 7.4 为什么这比“多试几次”更工程化

因为“多试几次”只是提高概率，不是建立可靠性。  
真正的工程化处理应该是：

`生成 -> 解析 -> 校验 -> 修复/重试 -> 决策执行`

而不是：

`生成 -> 看起来像 JSON -> 直接执行`

### 7.5 这一题怎么总结

**如果 JSON 输出错了，我的做法不是让模型瞎重来，而是优先避免自由 JSON，改用 Function Calling；如果必须用 JSON，就走 schema 约束、解析校验、repair、重试/回退这一整套容错链路。**

## 8. 这个项目为什么算多智能体？为什么不用一个大模型直接做完？多 Agent 的四种范式是什么？比单 Agent 有什么优势？

### 8.1 这个项目为什么可以算多智能体

因为它满足多 Agent 系统最关键的三个条件：

1. **角色分工明确**
2. **每个角色有自己的权限边界**
3. **角色之间通过消息协作完成任务**

当前项目里的角色非常清楚：

- `travel_router`：总控、意图分发、记忆读写、指代消解
- `weather_agent`：天气专家
- `spot_agent`：景点/攻略专家
- `plan_agent`：路线/酒店/行程专家

它们不是一个 prompt 里假装“我有多个身份”，而是实际以不同 agent 配置、不同 trigger、不同工具权限在运行。

### 8.2 为什么不让一个大模型直接全做

如果只从“能不能跑通 demo”的角度，当然一个大模型也能全做。  
但做工程系统时，单 Agent 往往会遇到四个问题。

#### 第一，Prompt 太重，职责边界不清

一个 Agent 既管天气、又管景点、又管路线、又管酒店、又管记忆、又管闲聊，prompt 会越来越长，规则互相干扰也越来越严重。

#### 第二，工具权限过大，容易越权

如果一个 Agent 拥有所有工具，它就更容易：

- 该分发时不分发
- 该先问清楚时直接乱查
- 该只查天气时顺便生成别的内容

#### 第三，评测和调优不容易切分

现在这个项目里我可以单独评估：

- Router 的路由是否对
- Spot 的回答是否对
- RAG 的召回是否对

如果全塞给一个 Agent，很多错误就难以定位。

#### 第四，后续微调和替换不灵活

现在 Router 可以单独上 Router LoRA，Spot 可以单独上 Spot LoRA。  
如果是一个大一统 Agent，就很难做角色级替换。

### 8.3 多 Agent 的四种范式，我会怎么答

这题严格说没有全行业唯一官方版本，所以我会先说明：

**如果按工程实践做一个常见分类，我会把多 Agent 范式概括成四种。**

#### 第一种：Router / Hub-and-Spoke 范式

一个中心 Agent 负责路由，其他专家 Agent 负责执行。  
当前项目主要就是这个范式。

#### 第二种：Planner-Executor 范式

一个 Agent 负责把大任务拆步骤，另一些 Agent 负责执行具体步骤。  
这类范式常见于复杂工作流和长任务规划。

#### 第三种：Blackboard / Shared Memory 范式

多个 Agent 不一定直接对话，而是围绕共享状态板或共享记忆读写协同。  
当前项目也有一点这个味道，因为存在结构化记忆，但主导仍然是 Router。

#### 第四种：Decentralized / Debate / Negotiation 范式

多个 Agent 地位相近，通过讨论、投票、博弈、辩论达成结果。  
这更适合复杂推理或多视角审校，但当前项目没有走到这一步。

### 8.4 多 Agent 相对单 Agent 的优势

我会总结成五点。

1. **职责更清晰**  
   每个 Agent 的 prompt 更短、更聚焦。

2. **更容易控权限**  
   Router 禁止查业务工具，Plan 禁止查景点工具，这种边界在多 Agent 里非常清楚。

3. **更容易评估和定位问题**  
   错在路由、检索、还是生成，比较容易拆开看。

4. **更容易做模块化微调**  
   Router LoRA 和 Spot LoRA 可以分开训练、分开换。

5. **更容易逐步扩展**  
   将来要加“美食 Agent”“多模态 Agent”，直接扩一个专家角色即可。

### 8.5 多 Agent 的代价我也会主动提一下

如果面试里只讲优点，容易显得不成熟。  
所以我会补一句代价：

1. 链路更长，延迟更高
2. 编排复杂度更高
3. 需要处理 Agent 间回声循环、消息冲突和状态同步

### 8.6 这题最后怎么收束

**这个项目之所以是多智能体，不是因为写了几个 YAML 文件，而是因为它真实存在角色拆分、消息协作和权限隔离。相比单 Agent，它的核心价值不只是“更高级”，而是更容易控制行为边界、做局部优化、定位问题和持续扩展。**

## 9. LoRA 数据集是怎么制作的？有没有 badcase 分析？

这题我会按“Router 数据、Spot 数据、评测集、badcase 结论”四部分回答。

### 9.1 Router LoRA 数据是怎么来的

Router 的训练目标不是写作文，而是学会：

- 什么时候读记忆
- 什么时候发给哪个 Agent
- 什么时候保存上下文
- 闲聊时什么时候自己回复

所以它的数据集本质上是**工具调用轨迹数据**。

仓库里有一个很关键的脚本：

- `tools/generate_finetune_data_v2.py`

这个脚本做的是高质量 tool-calling 数据生成，里面明确写了目标是生成多样化的工具调用训练数据，并且覆盖：

- 天气
- 景点
- 路线
- 行程
- 闲聊
- 指代消解
- 多轮追问
- 边界场景

它生成的是 ShareGPT 风格样本，里面包含：

- `human`
- `gpt`
- `tool_calls`
- `observation`

随后再通过：

- `tools/convert_toolcalls_to_function_call.py`

把 `gpt + tool_calls` 转成 LLaMA-Factory 更适合训练的：

- `function_call`
- `observation`

格式。

当前工作区里最终准备好的 Router 数据文件是：

- `data/lora_ready/01_tool_calling.json`
- `data/lora_ready/01_tool_calling_lf.json`

两者当前都是 `833` 条样本。

### 9.2 Router 数据为什么这样做

因为 Router 任务最重要的是**行为模式**，不是知识记忆。

它要学的是：

`用户输入 -> 读记忆 -> 识别意图 -> 选正确动作 -> 补正确参数`

所以最好的数据形式不是普通问答对，而是**完整动作链样本**。

### 9.3 Spot LoRA 数据是怎么来的

当前工作区里 Spot 的准备数据文件是：

- `data/lora_ready/02_spot_cultour_clean.json`

当前样本数是 `8802`。

从文件名、训练用途和周边脚本可以判断，它更偏向**景点/文旅问答与攻略表达数据**，不是纯工具调用数据。

因此 Spot 数据的目标和 Router 数据完全不同：

- Router 数据强调“动作对不对”
- Spot 数据强调“回答像不像一个景点助手”

这也正是为什么 Router LoRA 很适合做 tool-calling SFT，而 Spot LoRA 更容易出现“风格提升但 strict 指标下降”的现象。

### 9.4 评测集是怎么准备的

我不会只说“训完跑一下看看”，而是要说明我有独立评测集。

当前仓库可见的几个关键评测集包括：

- `tools/router_eval_cases.json`：`8` 条 Router 精准用例
- `tools/spot_eval_cases.json`：`80` 条 Spot 内部用例
- `tools/spot_eval_crosswoz.json`：`150` 条公开 CrossWOZ 基准用例
- `tools/expanded_test_cases.json`：`198` 条扩展测试用例

其中 `tools/build_router_eval_dataset.py` 和 `tools/build_spot_eval_from_crosswoz.py` 体现了一个很重要的工程意识：

**训练集和评测集不是一回事，评测必须独立。**

### 9.5 Router 的 badcase 主要是什么

根据已有 badcase 总结和评测结果，Router 的问题分成“基础模型问题”和“LoRA 后残留问题”。

#### 基础模型阶段

基础模型最典型的问题是：

1. 明明是 Router，却绕过系统设计，直接调用业务工具
2. 该给 `weather_agent` 的，却自己查天气
3. 该给 `plan_agent` 的，却自己乱答路线
4. 参数不完整，尤其是跨城、多轮场景

这其实说明：  
**基础模型虽然能理解提示词，但没有把“你只是路由器”这条规则学稳。**

#### Router LoRA 后

LoRA 之后，核心动作已经大幅变稳。  
剩余 badcase 更多集中在：

- `save_context` 字段漏填
- 对 `travel_party`、`notes`、`trip_days` 这类信息提取不完整
- 某些多轮条件补全还不够满

也就是说，问题从“动作错”升级成了“细节没抠满”，这是一个很明显的进步。

### 9.6 Spot 的 badcase 主要是什么

根据 `docs/lora真实badcase汇总.md` 和相关总结，Spot LoRA 典型问题在这些地方：

1. **游玩时长表达不稳**
2. **预算/门票类信息覆盖不完整**
3. **景点名称没有显式点出**
4. **更会写描述，但不一定更贴 benchmark 的槽位格式**

这和第 5 题的分析是一致的：  
它不是“完全不会答”，而是“描述性增强、结构性下降”。

### 9.7 我做 badcase 分析时看什么

我通常会把 badcase 分成三层。

#### 第一层：动作错没错

适用于 Router：

- 工具名错
- 参数错
- 链路错

#### 第二层：事实错没错

适用于 Spot / RAG：

- 该说的事实漏了没有
- 有没有把检索上下文没给的信息编出来

#### 第三层：格式错没错

有时候事实本身差不多，但格式不符合评测预期，也会掉 strict 指标。

### 9.8 这题最后怎么总结

**我做 LoRA 不是“找个数据集丢进去跑”。Router 数据是按工具调用链来构造和转换的，Spot 数据更偏景点/文旅问答。训练之外，我还配了独立评测集和 badcase 分析。Router badcase 从“经常动作错”收敛到了“少数字段没补全”；Spot badcase 则暴露出风格化 SFT 和严格槽位评测之间的矛盾。**

## 10. 这个项目的提示词工程/优化做了什么工作？

这题我会强调：  
**这个项目的 Prompt 工程不是“写几句漂亮话”，而是在设计 Agent 的行为协议。**

### 10.1 Router 的 Prompt 重点是“权限约束 + 工作流约束”

Router 的 instruction 里做了非常强的流程约束：

1. 必须先 `get_context`
2. 必须做指代消解
3. 天气/景点/行程/酒店只能转给专家 Agent
4. 禁止自己调用业务工具
5. 分发完不要自己再在频道里重复回答
6. 如果消息来自其他 Agent，要直接 `finish`
7. 出现新信息时要 `save_context`

这已经不是普通意义上的“提示词优化”，而是**把业务状态机写进 Prompt**。

### 10.2 Router Prompt 里最值钱的是示例驱动的边界学习

当前 Router prompt 里给了多个很具体的 few-shot 风格示例，例如：

- 天气查询怎么转发
- 跨城路线怎么保存 `origin_city/destination_city`
- 多日规划怎么补 `departure_time/trip_days`
- 酒店追问怎么用 `destination_city` 消解“那边”

这些示例的价值不在于“让回答更好看”，而在于：

**让模型学会什么时候提取什么字段，什么时候做哪种动作。**

### 10.3 Weather Prompt 的重点是“必须走真实天气工具”

天气 Agent 的 Prompt 做了几个关键限制：

1. 必须调用 `get_weather`
2. 只能处理来自 Router 的 direct message
3. 不要扩展到景点、路线等领域
4. 回复时必须用 `send_channel_message`

也就是说，它不是一个泛聊天助手，而是强工具型专家。

### 10.4 Spot Prompt 的重点是“检索策略显式化”

Spot Prompt 里最重要的优化，不是措辞，而是把检索决策写明白了：

- 城市级泛推荐：优先 `search_spots`
- 攻略/避坑/怎么玩：才调用 `search_local_knowledge`
- 如果两个都用，先 POI 再 RAG
- 如果记忆里有 `travel_party`、`preferences`，推荐时要考虑

这一步非常关键，因为如果不把“什么时候查 POI，什么时候查 RAG”写明，模型就容易乱用工具。

### 10.5 Plan Prompt 的重点是“路线/酒店专责化”

Plan Prompt 里我做的关键约束有：

1. 处理前必须 `get_context`
2. 跨城问题必须优先 `get_driving_route`
3. 酒店问题必须 `search_hotels`
4. 禁止调用 `search_spots`、`search_local_knowledge`、`search_combined`
5. 如果 `search_hotels` 返回了分组标签，要按分组总结

这使得 Plan Agent 不会因为“也懂一点景点”就越权乱搜。

### 10.6 为什么 Prompt 还重要，既然已经有 LoRA 了

这是一个很好的追问。

我会这样答：

- Prompt 决定运行时规则
- LoRA 决定模型对这些规则的内化程度

Prompt 像制度，LoRA 像训练。  
只有制度没有训练，模型容易偶尔不守规矩；只有训练没有制度，系统规则又不透明、不可维护。

### 10.7 这个项目里的 Prompt 优化，本质提升了什么

主要提升了三件事：

1. **边界清晰度**
2. **动作一致性**
3. **多轮上下文利用率**

也就是让 Agent 更像在执行协议，而不是即兴发挥。

### 10.8 这题最后怎么总结

**这个项目里的 Prompt 工程，本质是在给多 Agent 系统写“行为操作系统”。Router 的重点是权限和状态流转，Spot 的重点是检索策略，Plan 的重点是路线/酒店专责化，Weather 的重点是真实工具闭环。LoRA 不是替代 Prompt，而是让这些规则更稳定地被执行。**

## 11. 项目里提到 MCP，怎么做的？有什么作用？

### 11.1 先讲概念：MCP 在这里不是空泛名词，而是工具暴露协议

MCP 即 Model Context Protocol。  
在这个项目里，它的作用非常具体：

**把天气、景点、酒店、路线、记忆等能力，以标准化工具接口暴露给 Agent。**

也就是说，MCP 在这个项目里扮演的是“模型可调用工具层”。

### 11.2 当前项目里有两类 MCP Server

#### 第一类：`memory_mcp.py`

这是专门给 Router 用的轻量记忆 MCP，只暴露：

- `save_context`
- `get_context`

这样做的好处是：  
Router 读写记忆可以，但拿不到天气、路线、酒店这些业务工具权限，防止越权。

#### 第二类：`mcp_server.py`

这是综合旅游 MCP，当前暴露的工具包括：

- `get_weather`
- `search_spots`
- `search_hotels`
- `search_local_knowledge`
- `search_combined`
- `get_driving_route`
- `save_context`
- `get_context`

专家 Agent 通过它去查天气、景点、酒店、路线，也可以顺带读写记忆。

### 11.3 它是怎么接进 Agent 的

当前 YAML 配置里，Agent 都是通过 stdio 方式连接 MCP：

- Router 连接 `memory_mcp.py`
- Weather / Spot / Plan 连接 `mcp_server.py`

这意味着 Agent 在运行时会把这些 MCP 暴露出来的函数看成“可调用工具”。

### 11.4 MCP 在这个项目里具体有什么价值

我会总结成四点。

#### 第一，标准化工具接口

工具一旦用 MCP 暴露出来，模型侧看到的就是统一的函数接口，而不是东一块西一块脚本调用。

#### 第二，权限隔离

Router 只接 memory MCP，这就是权限控制。  
Plan 接综合旅游 MCP，但 prompt 又限制它不能碰景点工具，这就形成了“协议层 + prompt 层”的双重约束。

#### 第三，便于扩展

以后要加“美食搜索”“门票查询”“多模态识图”，可以继续往 MCP 层加工具，而不用把所有逻辑硬写进 Agent 本体。

#### 第四，便于和 Function Calling 对接

从 Agent 的视角看，MCP 暴露出来的就是可调用函数。  
所以它天然适合和 Function Calling 工作流结合。

### 11.5 举个当前项目里的真实例子

例如天气问题：

1. Router 识别天气意图
2. Router 发 direct message 给 `weather_agent`
3. `weather_agent` 调 `get_context`
4. `weather_agent` 调 `get_weather`
5. `weather_agent` 用 `send_channel_message` 回复

在这里，`get_context` 和 `get_weather` 都是 MCP 工具。

### 11.6 MCP 不是“让模型更聪明”，而是“让模型更能接系统”

这一句我觉得面试里很重要。

MCP 本身不提供智能，它提供的是：

- 标准化接入
- 能力边界
- 可组合性

也就是说，它解决的是**系统工程问题**，不是“参数里学会世界知识”的问题。

### 11.7 这题最后怎么总结

**在这个项目里，MCP 的作用是把工具能力标准化成模型可调用接口，并顺手完成权限隔离。Router 只接记忆 MCP，专家 Agent 接综合旅游 MCP。这样模型不是空口回答，而是能通过统一协议调外部真实能力。**

## 12. 项目无关：机器学习降维、PyTorch / Paddle、分类模型八股、反问、未来规划怎么答？

这题明显超出当前项目本身了，所以我会分成“我能结合项目答的”和“通用基础答法”两部分。

### 12.1 降维怎么答

#### PCA

PCA 的目标是找到投影方向，让投影后方差最大。  
如果样本协方差矩阵是 `S`，投影矩阵是 `W`，那么目标可以写成：

`max tr(W^T S W)`

约束通常是：

`W^T W = I`

解法上，本质就是对协方差矩阵做特征值分解，取最大特征值对应的特征向量。

#### SVD

对矩阵 `X` 做：

`X = U Σ V^T`

可以用于降维、压缩、去噪。  
在很多场景里，PCA 和 SVD 是密切相关的。

#### t-SNE / UMAP

这两类方法更适合可视化，不太适合做大规模线上特征压缩。

- t-SNE 更强调局部邻域保持
- UMAP 通常速度更快，兼顾局部与一定全局结构

#### AutoEncoder

如果是非线性降维，可以用自编码器：

`x -> encoder -> z -> decoder -> x_hat`

通过最小化重建误差，让中间的低维向量 `z` 成为压缩表示。

### 12.2 PyTorch / PaddlePaddle 了解多少，怎么答

我会诚实回答：

**当前这个项目实际用的是 PyTorch 生态，不是 Paddle 生态。**

## 13. 是否了解多模态 Agent 解决方案？

这题我会先如实说明现状，再讲扩展方案。

### 13.1 当前项目现状：本项目主线还是文本多 Agent，不是完整多模态系统

目前这个旅游助手项目的真实主链路是：

- 文本输入
- Router 分发
- 文本工具/RAG 检索
- 文本回复

所以如果面试官问“你这个项目是不是已经做了多模态”，我不会硬说是。  
更准确的回答是：

**当前项目主线还是文本多 Agent，但它的结构天然适合往多模态扩展。**

### 13.2 多模态 Agent 的核心思想是什么

多模态 Agent 的本质不是“接了图片就叫多模态”，而是：

**让系统能接收、理解、路由、使用不同模态的信息，并把这些信息整合进决策链路。**

例如输入可能来自：

- 图片
- 语音
- OCR 文档
- 地图截图
- 视频帧

而系统最终需要做的是：

- 识别图中实体
- 抽取文字
- 理解用户意图
- 再交给对应业务 Agent 处理

### 13.3 如果把当前项目扩成多模态，我会怎么设计

我会基于现有 Router 架构往前加一层“感知专家”。

比如可以新增：

1. **Vision/OCR Agent**
2. **ASR Agent**
3. **地图识别 Agent**

然后总体流程变成：

```text
用户上传图片/截图/语音
   -> 多模态预处理 Agent（OCR / VLM / ASR）
   -> 抽取文本、实体、地点、景点名、菜单名、路牌信息
   -> travel_router 做统一意图编排
   -> spot_agent / plan_agent / weather_agent 继续执行
```

### 13.4 旅游场景里有哪些真实多模态用例

这个问题如果能答得贴项目，会比较加分。

#### 场景一：拍景点图片问“这是哪里，值不值得去”

可以先让视觉模型做地标识别，再交给 `spot_agent` 查攻略和玩法。

#### 场景二：拍导航牌、车票、酒店订单截图

先 OCR 抽字，再让 Router 判断：

- 是路线问题
- 还是酒店问题
- 还是行程核对问题

#### 场景三：拍菜单或特产问“这是什么，推荐买吗”

可以扩一个美食/购物专家 Agent。

### 13.5 多模态 Agent 最难的地方是什么

我不会只说“模型更复杂”，而会说三件事：

1. **模态对齐**  
   图片里识别出的实体，怎么稳定映射到文本知识库里的景点名？

2. **误识别传染**  
   OCR/VLM 一旦前面识别错，后面 Router 和专家 Agent 都会跟着错。

3. **成本和延迟更高**  
   多模态前处理本身就会增加时延和成本。

### 13.6 这题最后怎么总结

**当前项目还不是完整多模态 Agent，但它的 Router + 专家架构非常适合扩多模态。实际落地时，可以先加感知类 Agent 把图片/语音转成结构化文本或实体，再接入现有的旅游专家链路。**

## 14. 你用的 OpenAgents 框架，讲讲这个框架的大致特点？为什么不用 LangChain？你了解 LangChain 吗？

### 14.1 这个项目里 OpenAgents 的实际特点是什么

我不会按官网宣传语答，而会按当前项目落地方式来答。

这个项目里 OpenAgents 最明显的特点有五个。

每个 Agent 的：

- `triggers`

- 让 Router 监听公共频道

这种问题更像“多 Agent 事件编排”，而不是简单的 chain。

#### 当前项目为什么优先 OpenAgents

因为当前项目更强调：

1. 多角色
2. 消息路由
3. 事件触发
4. Agent 权限边界

在这个前提下，OpenAgents 的形态和当前系统更贴。

#### LangChain / LangGraph 什么时候更合适

如果以后我要做的是：

- 强确定性的状态图
- 复杂 DAG 工作流
- 更细粒度的 node/edge 控制
- 检索链、工具链、状态机显式编排

我会诚实说：  
**了解基础能力和核心抽象，但这个项目没有直接基于 LangChain 重构。**

- Tools

- AgentExecutor
- LCEL
- LangGraph

- LangChain 擅长链式组件拼装
- LangGraph 擅长显式图编排和可控状态流

1. 有些行为逻辑仍然主要靠 prompt 约束，确定性还不够强

## 15. 记忆模块怎么做？为什么不把所有历史对话全塞进去？

### 15.1 当前项目的记忆模块是怎么做的

当前项目的记忆不是“全聊天记录数据库”，而是**结构化状态记忆**。

核心字段包括：

- `current_city`
- `origin_city`
- `destination_city`
- `current_spot`
- `travel_party`
- `preferences`
- `departure_time`
- `trip_days`
- `notes`
- `last_updated`

底层当前是写到：

- `storage/user_context.json`

记忆工具由：

- `tools/memory_tools.py`

提供实际读写；  
再通过：

- `memory_mcp.py`
- `mcp_server.py`

暴露给 Agent 调用。

### 15.2 当前链路里谁在用记忆

#### Router

Router 必须先 `get_context`，然后做：

- 指代消解
- 意图分发
- 新信息提取
- `save_context`

#### 专家 Agent

Weather / Spot / Plan 的 instruction 里也要求先 `get_context`，这样即使 Router 发来的文本里没有写全，专家也能结合历史状态补全。

### 15.3 为什么不把所有历史对话都塞进模型

这个问题非常重要，我会从三个角度答。

#### 第一，成本会越来越高

标准自注意力下，时间和显存开销会随序列长度快速增长。  
粗略看，注意力复杂度接近：

`O(n^2)`

如果历史越堆越长，prefill 时间会越来越重。

#### 第二，噪声会越来越大

不是所有历史对当前问题都有用。  
把几十轮闲聊和无关对话全塞进去，会稀释真正关键的信息。

#### 第三，状态型信息其实更适合结构化存储

比如：

- 当前在哪个城市
- 想去哪
- 带不带小孩
- 有几天时间

这些信息本质上是 state，不是必须每次都用原文重放。

可以理解成：

`s_t = f(s_{t-1}, x_t)`

也就是当前状态 `s_t` 是在旧状态 `s_{t-1}` 基础上，根据新一轮输入 `x_t` 更新出来的。  
既然能维护状态，就没必要每次都把整个原始轨迹重新塞给模型。

### 15.4 当前项目是怎么折中的

它不是完全不用历史，而是做了一个折中：

## 16. Function Calling 是怎么回事？你项目里提到了 Function Calling，具体是什么？

### 16.1 先讲概念

Function Calling 的核心不是“模型会写函数”，而是：

**让模型在合适的时候，输出结构化调用意图，由系统去执行对应工具。**

也就是说，模型不直接回答：

`“北京今天 18 度，多云。”`

而是先输出类似：

```json
{
  "name": "get_weather",
  "arguments": {
    "city": "北京"
  }
}
```

然后外部系统执行 `get_weather("北京")`，再把结果回给模型，或直接用于回复。

### 16.2 它和普通文本生成的本质区别

普通文本生成是在做：

`y = p_theta(. | x)`

Function Calling 则更像是在做：

`a = policy_theta(x)`

其中 `a` 是一个动作，包含：

- 调哪个函数
- 传什么参数

所以从工程角度看，Function Calling 本质是**结构化动作预测**。

### 16.3 当前项目里哪些地方体现了 Function Calling

这个项目里非常多地方都在体现。

#### Router

Router 本质上就在做动作选择：

- `get_context`
- `save_context`
- `send_direct_message`
- `send_channel_message`

它的核心任务不是自由回答，而是选对动作链。

#### Weather Agent

它要做的是：

1. `get_context`
2. `get_weather`
3. `send_channel_message`

#### Plan Agent

它要做的是：

1. `get_context`
2. `get_driving_route` 或 `search_hotels`
3. `send_channel_message`

#### Spot Agent

它要做的是：

1. `get_context`
2. `search_spots` / `search_local_knowledge`
3. `send_channel_message`

### 16.4 它和 MCP 的关系是什么

这两个概念容易混。

#### MCP

MCP 解决的是：

- 工具如何以标准协议暴露出来

#### Function Calling

Function Calling 解决的是：

- 模型如何决定调用哪个工具、传什么参数

所以可以理解成：

- MCP 是工具接入层
- Function Calling 是模型动作层

两者在这个项目里是配套关系。

### 16.5 为什么 Function Calling 对这个项目特别重要

因为这个项目很多任务都不适合“让模型瞎答”。

比如：

- 天气
- 路线
- 酒店
- 记忆读写
- 任务分发

这些都是动作型任务。  
动作型任务最适合的不是长篇自由文本，而是结构化调用。

### 16.6 当前项目里 Router LoRA 为什么提升这么大

因为 Router 这个任务几乎就是 Function Calling 学习题。

当我们把 Router 训练成更稳定地预测：

- 该不该 call tool
- call 哪个 tool
- 参数怎么填

它的业务成功率自然就会大幅提升。

这也是为什么 Router LoRA 的 benchmark 提升会远比“语言风格改善”更明显。

### 16.7 如果面试官追问“Function Calling 和 JSON 输出有什么关系”

我会这样答：

- JSON 输出只是结构化格式
- Function Calling 是一个更高层的动作机制

也就是说，Function Calling 的底层可以用 JSON 表达参数，但它的本质不是“输出一段 JSON”，而是“触发一个受约束的系统动作”。

### 16.8 这题最后怎么总结

**Function Calling 本质是让模型从“说话”转成“做动作”。在这个项目里，Router、Weather、Spot、Plan 都在做不同形式的函数调用决策；MCP 负责把工具接进来，Function Calling 负责让模型选对工具和参数。**

## 17. SFT 这种垂域微调的意义和价值在哪里？

这题我不会答成“让模型更懂领域”，因为这个说法太笼统。

### 17.1 SFT 的本质是什么

SFT 的训练目标通常写成：

`L_SFT = - Σ_{t=1}^T log p_theta(y_t | x, y_<t)`

也就是说，它做的是**在给定输入条件下，提高参考输出序列的条件概率**。

从工程角度看，SFT 最擅长优化的是：

1. 输出格式
2. 角色风格
3. 任务流程
4. 领域表达习惯

而不是凭空给模型“注入一个永远不会忘的知识库”。

### 17.2 这个项目里，SFT 的价值体现在哪

我会分 Router 和 Spot 两个角度讲。

#### Router 侧

Router LoRA 的价值非常明确：  
它让模型从“知道规则”变成“更稳定执行规则”。

也就是把这些行为学进参数：

- 先读记忆
- 天气转 `weather_agent`
- 景点转 `spot_agent`
- 路线/酒店转 `plan_agent`
- 新信息要存 `save_context`

这类能力如果只靠 prompt，模型可能“偶尔记得、偶尔不记得”；  
做了 SFT/LoRA 之后，动作分布会稳定很多。

#### Spot 侧

Spot LoRA 的价值也不是零。  
它证明了 SFT 能明显改变：

- 回答风格
- 旅游文案组织方式
- 自然语言表达相似度

只是它同时也提醒我们：  
**如果数据和指标没对齐，SFT 也可能把模型往“更会说，但不一定更准”的方向推。**

### 17.3 SFT、RAG、工具各自解决什么问题

这是面试里我很愿意主动讲清楚的一点。

#### 工具

解决的是**实时外部事实获取**。

比如：

- 天气
- 路线
- 酒店

#### RAG

解决的是**外部知识检索补充**。

比如：

- 景点攻略
- 手写避坑经验
- 深度玩法

#### SFT / LoRA

解决的是**模型如何稳定地使用这些能力**。

比如：

- 什么时候该路由
- 什么时候该调工具
- 结果拿到后怎么组织成目标风格

所以它们不是互相替代关系，而是分工关系。

### 17.4 为什么垂域微调有意义

因为通用大模型虽然“什么都懂一点”，但在具体业务里往往会出现三个问题：

1. 角色边界不稳
2. 输出格式不稳
3. 对特定工作流不稳

垂域 SFT 的意义就在于把这些“稳定性”学进去。

对于当前项目而言，最有价值的不是让模型背更多旅游知识，而是让它稳定遵循：

- Router 工作流
- 专家 Agent 工作流
- 旅游助手表达风格

### 17.5 为什么不是所有场景都值得 SFT

这题我会顺带体现一点判断力。

如果问题只是：

- 知识缺失
- 数据实时性要求高

那优先应该补工具或 RAG，而不是盲目微调。

只有当问题是：

- 行为模式不稳
- 输出格式不稳
- 角色边界不稳

SFT 才最有价值。

### 17.6 这题最后怎么总结

**SFT 的价值不在于替代知识库和工具，而在于把“角色、流程、格式、风格”固化进模型参数里。这个项目里 Router LoRA 的明显提升，就是 SFT 价值最直接的证据；Spot LoRA 的 mixed result 也反过来说明，SFT 必须和真实业务指标对齐。**

## 18. 讲解一下 Attention 机制？还有别的注意力计算方式吗？K、V 维度不同怎么办？padding、裁剪、维度变换怎么处理？

这题我会先讲标准 Attention，再讲工程细节。

### 18.1 标准 Scaled Dot-Product Attention

给定输入表示 `X`，先通过线性映射得到：

`Q = X W_Q`

`K = X W_K`

`V = X W_V`

然后计算注意力：

`Attention(Q, K, V) = softmax(Q K^T / sqrt(d_k)) V`

这里：

- `Q`：当前 token 想去关注什么
- `K`：每个 token 提供什么“可被匹配的索引”
- `V`：真正被取回和加权汇总的信息内容

### 18.2 为什么要除以 `sqrt(d_k)`

因为如果 `d_k` 很大，`QK^T` 的数值方差会变大，softmax 容易进入过尖或过饱和区域，训练不稳定。

除以：

`sqrt(d_k)`

相当于做一个尺度归一化，让 softmax 输入分布更稳定。

### 18.3 Multi-Head Attention 是什么

多头注意力本质上就是做多组不同的投影：

`head_i = Attention(X W_Q^(i), X W_K^(i), X W_V^(i))`

然后把各个头拼接：

`MultiHead(X) = Concat(head_1, ..., head_h) W_O`

好处是：

- 不同 head 可以关注不同关系
- 有的 head 看局部依赖
- 有的 head 看长程依赖
- 有的 head 可能更偏句法
- 有的 head 可能更偏实体对齐

### 18.4 还有别的注意力计算方式吗

有，而且不少。

#### 加性注意力（Bahdanau Attention）

不是直接点积，而是把 `Q` 和 `K` 过一个小网络后打分。  
早期 seq2seq 里比较常见。

#### 余弦注意力

按余弦相似度做匹配：

`cos(q, k) = (q · k) / (||q|| ||k||)`

#### 稀疏注意力

不是所有 token 两两都算，而是只算局部窗口或稀疏连接，减少复杂度。

#### 线性注意力

通过核技巧或重排计算，把复杂度从近似 `O(n^2)` 降到更低量级，适合超长序列。

#### MQA / GQA

这类方法会共享或分组共享 K/V，主要是为了加速推理、减少 KV cache。

### 18.5 K、V 维度不同怎么办

这个问题要分清楚“序列长度不同”和“特征维度不同”。

#### 情况一：序列长度不同

在 cross-attention 里很正常。

例如：

- Query 来自 decoder
- Key/Value 来自 encoder

这时：

- `len(Q)` 可以和 `len(K), len(V)` 不同
- 但 `K` 和 `V` 的序列长度要一致，因为每个 key 对应一个 value

#### 情况二：最后一维不同

真正做点积时，要求 `Q` 和 `K` 的最后一维一致，也就是都映射到 `d_k`。

所以通常会先做线性投影：

`Q = X_q W_Q,  Q ∈ R^(n_q x d_k)`

`K = X_k W_K,  K ∈ R^(n_k x d_k)`

`V = X_v W_V,  V ∈ R^(n_k x d_v)`

这里可以看到：

- `Q` 与 `K` 的最后一维必须相同，才能做 `QK^T`
- `V` 的 value 维 `d_v` 可以和 `d_k` 不同

所以如果原始输入维度不同，不是直接硬算，而是先靠线性层映射到可计算空间。

### 18.6 padding 怎么处理

padding 的核心原则是：

**补出来的 token 不能参与有效注意力。**

做法通常是加 mask，把 padding 位置在 softmax 前变成一个极小值：

`score_ij = -inf`

这样 softmax 后这些位置的权重接近 0。

如果不 mask，模型就会把注意力错误分配到补齐位上。

### 18.7 裁剪怎么处理

如果序列太长，常见处理有：

1. 直接截断
2. 滑动窗口
3. 分块检索后再聚合
4. 用长上下文模型或稀疏/线性注意力

在工程里，裁剪不是随便砍，而是要尽量保留：

- 最近上下文
- 系统指令
- 关键实体
- 结构化状态

这也正是为什么当前项目选择“结构化记忆 + 最近上下文”，而不是无脑塞满历史。

### 18.8 维度变换怎么理解

很多人把注意力看得太神秘，其实很大一部分工作就是线性代数里的维度变换。

例如：

- 原始输入维度是 `d_model`
- 每个头里的 `Q/K/V` 映射到 `d_k, d_k, d_v`
- 多头拼接后再映回 `d_model`

这些投影层就是模型学出来的“怎么看、怎么索引、怎么取值、怎么合并”。

### 18.9 这和 LoRA target q/k/v/o 有什么关系

这题我会顺势连回前面的 LoRA。

因为：

- `q_proj` 改的是“我看什么”
- `k_proj` 改的是“别人怎么被我匹配到”
- `v_proj` 改的是“我取回什么内容表示”
- `o_proj` 改的是“多头结果怎么合并回主干”

所以 LoRA 常加在这些层上，是因为它们正处在注意力机制的核心控制点。

### 18.10 这题最后怎么总结

**Attention 的本质是用 Query、Key 做匹配，用匹配权重加权汇总 Value。工程上最重要的是理解缩放、mask、维度投影和多头机制。K、V 维度不同并不可怕，关键是让 Q 和 K 投影到同一匹配空间，并保证 K、V 在序列维上对齐。**
