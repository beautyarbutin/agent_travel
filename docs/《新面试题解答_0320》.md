# Agent项目相关问题

## 1. 怎么解决幻觉，输出慢是什么原因以及如何解决？先说常见的方案，再举例说明你这个项目是怎么做的。

### 【面试当场如何回答】

- **通用答法**：我会先把幻觉分成两类，一类是事实型幻觉，一类是行为型幻觉。事实型幻觉一般靠工具调用、RAG、检索增强和输出约束解决；行为型幻觉一般靠角色拆分、权限隔离、流程约束和评测闭环解决。输出慢通常来自长上下文、模型解码、外部工具、检索和多步编排。
- **结合我这个项目**：我这个项目主要用了四层方法压幻觉：天气和路线这类强事实问题必须走工具；`travel_router` 和专家 Agent 做权限隔离；景点问题用 `高德 POI + 本地 RAG` 做 grounding；再用 Router tool-calling benchmark、Spot benchmark 和 RAG 检索评测去找 badcase。速度方面，我主要从结构化记忆、减少无效检索、懒加载索引和缩短链路这些方向优化。

### 【详细解释】

#### 1.1 幻觉为什么会出现

- **事实型幻觉**：本来应该查工具或知识库的数据，模型直接编。
- **行为型幻觉**：本来只该路由的 Agent 越权自己查业务工具，或者该发消息却吐纯文本。

#### 1.2 常见解决思路

| 问题类型 | 常见方案 |
| --- | --- |
| 事实型幻觉 | Function Calling、外部工具、RAG、检索增强、输出格式约束 |
| 行为型幻觉 | 多 Agent 角色拆分、权限隔离、状态机/Prompt 约束、benchmark + badcase |

#### 1.3 我这个项目怎么做

1. **高风险事实问题强制走工具**
   - 天气：`get_weather` / `get_weather_forecast`
   - 路线：`get_driving_route`
2. **权限隔离**
   - `travel_router` 只做 `get_context / save_context / send_direct_message`
   - `weather_agent / spot_agent / plan_agent` 才能做业务工具调用
3. **景点场景用 Hybrid RAG**
   - 高德 POI 负责实时地点信息
   - 本地 RAG 负责攻略、避坑、玩法
4. **评测闭环**
   - Router：tool-calling benchmark
   - Spot：strict success / fact coverage / ROUGE-L
   - RAG：Hit@K / MRR

#### 1.4 输出慢的来源

可以把总时延近似拆成：

`T_total ≈ T_prefill + T_decode + T_tool + T_network + T_retrieval + T_orchestration`

| 慢的来源 | 项目里的表现 | 优化思路 |
| --- | --- | --- |
| 上下文太长 | 历史消息越多，prefill 越慢 | 用结构化记忆替代整段原文重放 |
| 外部 API | 天气和 POI 都有网络请求 | 缓存热点城市、减少重复查 |
| RAG 检索 | FAISS + embedding + 融合排序 | 懒加载、缓存热门查询 |
| 多 Agent 编排 | Router 要先分发，专家再执行 | 简单问题走最短路径 |

## 2. agent间数据怎么流动【节点间怎么运行的】，你的项目里边举个例子？比如travel_router 是如何进行意图分发，用户提问天气问题后交给weather agent的？消息路由机制具体怎么做的？

### 【面试当场如何回答】

- **通用答法**：多 Agent 的数据流一般包括三层：自然语言消息流、结构化状态流、工具结果流。最常见的机制是“统一入口识别意图，再把任务路由给对应专家节点”。
- **结合我这个项目**：我这个项目不是广播抢答，而是 `Router + Direct Message 专家 Agent`。用户先在公共频道发消息，`travel_router` 监听公共频道，先 `get_context`，做指代消解和意图识别，然后通过 `send_direct_message(target_agent_id="weather_agent", text=...)` 转给 `weather_agent`。`weather_agent` 只监听来自 Router 的 direct message，再执行 `get_context -> get_weather -> send_channel_message`。

### 【详细解释】

#### 2.1 运行机制

| 环节 | 谁负责 | 事件类型 |
| --- | --- | --- |
| 公共频道入口 | `travel_router` | `thread.channel_message.notification`、`thread.reply.notification` |
| 定向执行 | `weather_agent / spot_agent / plan_agent` | `thread.direct_message.notification` |

#### 2.2 三类数据在流动

1. **自然语言任务数据**：用户原问题、Router 转发后的文本。
2. **结构化状态数据**：`current_city`、`current_spot`、`preferences` 等。
3. **工具返回事实数据**：天气、POI、路线结果。

#### 2.3 天气问题的完整链路

```text
用户在 general 发问
-> travel_router 收到消息
-> get_context 读取记忆
-> 判断为天气意图
-> send_direct_message 给 weather_agent
-> 视情况 save_context(current_city=...)
-> weather_agent 收到 direct message
-> get_context 补全上下文
-> get_weather(city=...)
-> send_channel_message 回 general
```

#### 2.4 当前消息路由机制为什么能工作

- Router 和专家 Agent 的 trigger 不同
- Router 有发送者过滤，忽略来自其他 Agent 的消息
- 专家 Agent 只处理来自 Router 的 direct message

这避免了“Agent 听到别的 Agent 回复又重新处理”的回声循环。

## 3. 你提到lora微调，这个lora微调和直接修改提示词、RAG 这两种工作有什么效果上的区别？你怎么弄的测试指标衡量lora效果？ 你是做lora微调，还是就学了个lora原理呢、讲讲lora原理？超参数的数值如何确定，列出你lora时候的超参数选定。然后全参训练、qlora和 lora的 区别 优劣 选择？你项目中lora训练取哪个step？我看你有个lora取得不是final而是中间一个checkpoint 为什么？数据处理/参数选择 step选择 损失分析 这一块 就是你是有一个评价指标，你的目的是提升模型在这上面的性能 有一个目标，针对这个目标你做了啥。lora为什么rank要设置成8，16这样的数，数字大小除了对显存占用有影响，对模型效果的影响真的这么大吗？为什么alpha大家都是设置成rank的两倍，这个的原因是什么呢，原论文不是相等吗？还有不同的target_module在模型微调中具体起了什么作用。加在qkvo这几个上有什么区别。 你的lora数据集是怎么制作的？lora微调的badcase分析有没有？lora为啥省显存？

### 【面试当场如何回答】

- **通用答法**：Prompt、RAG、LoRA 解决的问题不一样。Prompt 是把规则告诉模型，RAG 是补外部知识，LoRA 是把角色行为和输出分布更稳定地学进参数。LoRA 的核心是冻结大部分原模型参数，只训练一个低秩增量，所以显存和训练成本都比全参小。
- **结合我这个项目**：我不是只学了 LoRA 原理，而是实际做了两条 LoRA：`router_lora` 和 `spot_lora`。`router_lora` 的目标是提升路由和工具调用稳定性，评测用的是 tool-name accuracy、argument exact match、turn exact match、sample full-chain success；结果里完整链路成功率从 `20.59%` 提升到 `85.29%`。`spot_lora` 更偏回答风格增强。超参数上两条线都用了 `r=8, alpha=16, dropout=0.05`，Router 学习率 `5e-5`，Spot 学习率 `3e-5`。Router 最后取的是 `checkpoint-60`，因为下游 benchmark 比 final 更好。

### 【详细解释】

#### 3.1 Prompt、RAG、LoRA 的区别

| 方案 | 改什么 | 解决什么问题 | 局限 |
| --- | --- | --- | --- |
| Prompt | 输入条件和规则 | 告诉模型该怎么做 | 模型不一定真的学住 |
| RAG | 外部知识输入 | 补知识、降知识幻觉 | 不直接解决角色边界 |
| LoRA | 模型参数增量 | 稳定角色行为、格式、动作分布 | 数据和指标不对齐时也会学偏 |

你可以直接背一句：

**Prompt 像制度，RAG 像查资料，LoRA 像把工作习惯训练进参数。**

#### 3.2 LoRA 原理

LoRA 的核心写法：

`W' = W0 + ΔW`

`ΔW = (alpha / r) * B A`

其中：

- `W0`：原始权重
- `A`、`B`：低秩矩阵
- `r`：rank
- `alpha`：缩放系数

为什么省显存：

1. 不更新原模型大参数，只训练小的 adapter 参数。
2. 优化器状态只为 adapter 保存，不为全模型保存。
3. 存储时只需要保存 adapter，而不是整套模型权重。

#### 3.3 LoRA / QLoRA / 全参训练怎么比较

| 方案 | 优点 | 缺点 | 适合什么场景 |
| --- | --- | --- | --- |
| 全参微调 | 容量最大、可塑性最强 | 显存高、训练慢、易破坏通用能力 | 大资源、任务边界明确 |
| LoRA | 显存友好、训练快、adapter 可插拔 | 容量有限 | 角色化微调、快速迭代 |
| QLoRA | 在 LoRA 基础上再量化底座，更省显存 | 训练链路更复杂、量化细节要注意 | 显存更紧张的场景 |

当前项目为什么优先 LoRA：

- 任务是角色化行为适配，不是全面重塑模型
- 需要分角色做 adapter
- 需要快速迭代 benchmark

#### 3.6 为什么 `rank` 常从 8、16 开始

- 不是玄学
- 是容量、泛化、显存之间的工程折中
- `r` 太小容易学不动，`r` 太大又不一定继续提升

`r` 影响的不只是显存，也影响模型能学到多复杂的更新子空间。  
新增参数量近似：

`Params_LoRA ≈ r * (d_in + d_out)`

#### 3.7 为什么很多人设 `alpha = 2r`

- 关键不是看 `alpha` 单独多大，而是看 `alpha / r`
- `alpha = 2r` 相当于 `scale = 2`
- 这是工程上常见的稳妥经验，不是必须遵守的定律

更严谨的说法是：

**原论文不是规定永远相等；后续工程实践里，很多人发现 `alpha / r` 设成 1 到 2 比较稳，所以 `2r` 变成常见习惯。**

| 数据线 | 样本量 | 目标 |
| --- | ---: | --- |
| Router | `833` | 工具调用轨迹学习 |

Router 数据处理链路：

1. 先生成 ShareGPT 风格 `gpt + tool_calls`
2. 再转成 LLaMA-Factory 用的 `function_call + observation`
3. 训练后如果要给 LM Studio 推理，再转回 `gpt + tool_calls`

这里要特别会解释“为什么要转格式”：

- 原始 `ShareGPT` 风格更像 OpenAI 聊天记录，模型说“我要调这个工具”
- 但 `LLaMA-Factory` 训练 tool-calling SFT 时，更稳定吃的是：
  - `human`
  - `function_call`
  - `observation`
  - 最后一条可学习的 `gpt`

在你这个项目里，原始 Router 数据 `01_tool_calling.json` 长这样：

- `human`: 用户问题
- `gpt + tool_calls`: 模型决定调哪个工具
- `observation`: 工具返回结果

例如：

- 用户：`洛阳会不会下雨`
- `gpt.tool_calls`：`get_context`
- `observation`：`暂无上下文记忆`
- `gpt.tool_calls`：`send_direct_message(weather_agent, ...)`
- `observation`：`消息已发送给weather_agent`

后来你用 `tools/convert_toolcalls_to_function_call.py` 把它改成：

- `human`
- `function_call`
- `observation`
- `function_call`
- `observation`

也就是说：

- 原来的 `gpt + tool_calls`
  更像“聊天消息里夹着工具意图”
- 后来的 `function_call + observation`
  更像“把动作和工具回执显式拆开”

面试里可以直接这样说：

> 我原始数据先按 ShareGPT 风格存成 `gpt + tool_calls`，因为它更接近 OpenAI 聊天接口的轨迹表达。后来为了让 LLaMA-Factory 更稳定识别成 tool-calling 监督样本，我又把它转成了 `function_call + observation`。这里 `function_call` 表示模型预测的动作，`observation` 表示工具执行后的返回结果。这个转换的本质，不是改业务语义，而是把同一条动作链换成更适合训练框架消费的表示法。

## 4. 加了个东西，反而导致模型效果变差了是什么原因？我看你这个spot lora有个指标还差了一些 那就分析不足，为什么没别人的好，哪里出了问题，数据量太少，还是数据质量太差，还是超参，还是模型结构太简单还是太复杂，欠拟合还是过拟合，改进思路是什么？

### 【面试当场如何回答】

- **通用答法**：模型加了新东西后变差，通常不是一句“训练失败”能解释，要区分训练目标和评测目标是否一致、数据分布是否匹配、模型是不是把容易优化的部分学得太重了。
- **结合我这个项目**：`spot_lora` 不是全面变差，而是出现了结构性变化。公开评测里 `ROUGE-L F1` 明显上升，说明更像旅游助手；但 `Strict success` 和 `Fact coverage` 下降，说明它更会说了，但不一定更像严格槽位抽取器。根本原因是训练目标、数据分布和评测指标没完全对齐。后续改进方向是把“事实抽取”和“自然表达”拆开。

### 【详细解释】

#### 4.1 原始结果怎么解读

| 指标 | Base | Spot LoRA | 结论 |
| --- | ---: | ---: | --- |
| Strict success | `98.00%` | `78.00%` | 下降 |
| Fact coverage | `99.00%` | `89.11%` | 下降 |
| ROUGE-L F1 | `2.66%` | `27.36%` | 明显上升 |

#### 4.2 为什么会这样

1. **训练目标和评测目标不一致**
   - SFT 优化 token 概率
   - strict success 看的是离散槽位全对
2. **数据分布更偏攻略表达**
   - `spot` 数据更像“会写旅游话术”
   - 不完全是“字段抽取数据”
3. **任务本身太杂**
   - 同时想要检索理解、事实抽取、自然表达

#### 4.3 这到底算过拟合还是欠拟合

不要简单二选一。更准确的说法是：

- 对风格分布来说，学得有点重
- 对结构化精度来说，约束又不够强

所以更像：

**风格过拟合 + 槽位约束不足**

#### 4.4 改进思路

| 改法 | 作用 |
| --- | --- |

#### 5.3.1 为什么我这个项目选 `bge-small-zh-v1.5`

如果面试官追问“为什么不是别的 embedding 模型”，最稳的回答不是背榜单，而是先回到你的系统约束。

我这个项目当时的真实约束是：

- 本地已经在跑 `Qwen3.5-4B + LM Studio`
- 机器资源紧，不能再额外依赖一个重型 embedding 服务
- 希望 RAG 能纯本地重建，不依赖远程 API
- 知识库规模是 `1.5 万+` 条，向量检索要能在本地稳定跑起来

在这个约束下，`bge-small-zh-v1.5` 的好处非常具体：

| 优点 | 结合我项目怎么理解 |
| --- | --- |
| 中文向量检索能力够用 | 你的问题大多是中文旅游问答，不需要先上多语种重模型 |
| 体量轻 | 文档里明确记录它大约 `90MB`，纯 CPU 也能跑 |
| 512 维向量 | 和你当前 `FAISS IndexFlatIP` 配套，建索引和查询成本都比较低 |
| 本地部署简单 | `SentenceTransformer(\"BAAI/bge-small-zh-v1.5\")` 直接可用，适合 MVP 和本地重建 |
| 摆脱远程 embedding API | 这是你这轮 RAG 改造的核心目标之一 |

所以更准确的说法是：

- 我不是先看谁排行榜最高
- 而是先看 **中文效果是否够用、CPU 能不能跑、能不能和现有 FAISS 链路直接接上**

一句现场回答可以这样说：

> 我最后选 `bge-small-zh-v1.5`，核心不是追最强，而是追“本地可落地”。我的机器本地已经在跑 `Qwen3.5-4B`，如果再叠一个重型 embedding 服务，链路会变得很重。`bge-small-zh-v1.5` 中文效果够用、模型轻、纯 CPU 可跑，和我现在 `FAISS + 1.5 万条景点知识库` 的规模是比较匹配的。

#### 5.3.2 和其他常见 embedding 模型怎么比

这个问题最适合用“和我当前场景的匹配度”来回答，而不是空讲优缺点。

| 模型 | 优点 | 缺点 | 和我项目的匹配度 |
| --- | --- | --- | --- |
| `bge-small-zh-v1.5` | 中文检索够用、轻量、CPU 可跑、接入简单 | 上限不如更大模型 | **最高，当前主选** |
| `bge-base-zh-v1.5` | 一般会比 small 更强一些，中文能力更足 | 更大、更慢、资源占用更高 | 能用，但对你本地部署性价比没那么好 |
| `bge-m3` | 更强，支持 dense / sparse / multi-vector，多语种能力也更强 | 更重，链路更复杂，通常更适合更完整的 IR 系统 | 对你这个中文旅游单机场景偏重 |
| 远程 embedding API | 效果可能更稳定，省本地部署心智 | 有网络依赖、成本依赖、服务依赖 | 和你这轮“完全本地化”目标相冲突 |

结合你的项目，实际可以这样理解：

1. `bge-base-zh-v1.5`
   适合“我愿意用更高资源换一点效果”的情况。  
   但你当前重点不是冲 embedding 榜，而是让整套 OpenAgents + 本地 RAG + LM Studio 真正稳定跑起来，所以 `small` 的性价比更高。

2. `bge-m3`
   它更适合更复杂的检索系统，比如：
   - 多语种
   - 长文档
   - dense + sparse + multi-vector 一体化
   - 后面还要接 rerank

   但你当前项目：
   - 主要是中文
   - chunk 是结构化景点记录，不是超长文档
   - 当前没有独立 reranker
   - 更重视本地可运行和工程简洁

   所以 `bge-m3` 对你当前阶段有点“性能过剩、工程过重”。

3. 远程 embedding API
   你项目里其实踩过这个坑。  
   之前本地切到 `LM Studio` 后，只跑了生成模型，没有单独跑 embedding 服务，原来的远程 / OpenAI-compatible embedding 链路就失效过。  
   所以这一轮你选本地 `bge-small-zh-v1.5`，本质上是在解决：

   - 本地链路可复现
   - 不依赖额外在线服务
   - 避免 embedding 接口和生成接口混线

一句话总结这个对比：

> `bge-small-zh-v1.5` 不是最强 embedding，但它是最符合我这套“中文旅游场景 + 1.5 万条知识库 + FAISS + 本地 CPU + 单机部署”约束的 embedding 模型。`bge-base` 是更大的同路线升级版，`bge-m3` 是更重、更复杂、更偏完整检索系统的方案，而远程 API 则和我这次本地化改造目标不一致。

#### 5.4 chunk 怎么切

当前最真实的说法：

- 一条结构化景点/攻略记录就是一个 chunk
- 不是 token 滑窗切块
- 也不是语义自适应切分

优点：

- 城市、景点名、标签不会被切碎
- 结构完整、好解释

缺点：

- 长文本的局部精准定位能力有限

#### 5.5 检索和融合怎么做

当前融合公式可近似写成：

`score(d, q) = 5 * dense_score + bm25_like_score + handwritten_bonus`

其中：

- `dense_score`：FAISS top-5 向量召回分
- `bm25_like_score`：查询时现算的启发式词法分
- `handwritten_bonus`：手写攻略命中时额外加权

#### 5.6 BM25 和“另一个检索”怎么融合

“另一个检索”就是 dense retrieval，也就是 embedding + FAISS。  
所以这里的融合不是“两个标准检索器”，而是：

- 语义召回分
- 词法补分
- 手写攻略 bonus

#### 5.7 rerank 用没用

严格说：

- **没有独立 cross-encoder reranker**
- 当前更像工程化融合排序

#### 5.8 远程 API 和本地 RAG 的优先级

稳妥说法不是“永远先谁后谁”，而是：

- 高德 POI：负责实时地点信息
- 本地 RAG：负责攻略、避坑、玩法
- 当前 `spot_agent` 往往会综合两路结果回答

#### 5.9 怎么排查 RAG

我会按五层定位：

1. query 有没有被错改写
2. 召回 topK 里有没有目标
3. 排序有没有把好结果压下去
4. 检索到了但生成没说对
5. 数据源本身有没有

#### 5.10 怎么评估

当前离线指标：

- `Hit@1 / Hit@3 / Hit@5`
- `MRR`
- 严格 Doc-ID 口径
- 宽松实体级口径

现成主报告结果基于 `15240` 条旧库版本：

| 口径 | Hit@1 | Hit@3 | Hit@5 | MRR |
| --- | ---: | ---: | ---: | ---: |
| 严格 Doc-ID | `35.9%` | `36.9%` | `36.9%` | `0.3636` |
| 宽松实体级 | `42.9%` | `43.9%` | `43.9%` | `0.4343` |

结论：

- 手写攻略很强
- 城市级问题较稳
- 精确景点 strict 检索仍是短板

#### 5.11 hallucination 有没有单独指标

严格说没有独立统一的 hallucination 总表。  
当前是通过三组指标间接看：

1. RAG 检索指标
2. Spot 回答指标
3. 人工 badcase 复盘

## 13. Function Calling  是怎么回事？你这个项目提到了Function Calling ，具体是什么 讲讲？

### 【面试当场如何回答】

- **通用答法**：Function Calling 的本质不是“模型会写函数”，而是让模型从“说话”转成“做动作”，输出结构化调用意图，让系统执行对应工具。
- **结合我这个项目**：我这个项目里 Router、Weather、Spot、Plan 都在做不同形式的 Function Calling / Tool Calling。Router 要决定是 `get_context`、`send_direct_message` 还是 `save_context`；Weather 要决定什么时候调 `get_weather`；Spot 要决定查 POI 还是查本地 RAG。这里我更愿意把它叫“结构化动作预测”，因为模型的任务不是组织一段漂亮回答，而是先选对动作、再把参数填对。

### 【详细解释】

普通文本生成 vs Function Calling：

| 方式 | 模型做什么 |
| --- | --- |
| 普通生成 | 直接给自然语言答案 |
| Function Calling | 先输出“调哪个函数、带什么参数” |

#### 13.1 Tool Calling 和 Function Calling 是一回事吗

在很多场景里，这两个词可以近似理解成一回事，但严格说：

| 叫法 | 更强调什么 |
| --- | --- |
| Function Calling | 更强调“调用一个结构化函数/接口” |
| Tool Calling | 更强调“调用一个外部能力”，不一定真是代码函数 |

在你这个项目里，`Tool Calling` 这个说法其实更贴切一点，因为模型调用的不只是“代码函数”，还包括：

- 查询天气
- 查询高德 POI
- 查本地 RAG
- 读写记忆
- 给别的 Agent 发 direct message
- 往频道发消息

所以这里最稳的说法是：

> Function Calling 是一种更常见的术语，Tool Calling 是更广义的工程说法。在我这个项目里，两者可以近似理解成同一类能力，本质都是让模型输出结构化动作，而不是自由文本。

#### 13.2 它和 MCP 是什么关系

这三个概念最好分开：

| 概念 | 作用 |
| --- | --- |
| MCP | 把工具接进系统 |
| Tool / Function Calling | 让模型决定调哪个工具、带什么参数 |
| observation | 工具执行后的返回结果 |

所以一条完整链路其实是：

`MCP 暴露工具 -> 模型做 tool calling -> 系统执行工具 -> observation 返回 -> 模型继续下一步`

在你项目里可以举最典型的 Router 例子：

`get_context -> send_direct_message -> save_context`

这里：

- `get_context`、`send_direct_message`、`save_context` 是工具
- 模型要学会什么时候调哪个工具
- 每次工具执行后的结果，就是下一轮决策要看的 `observation`

当前项目里的动作链：

| Agent | 核心动作链 |
| --- | --- |
| Router | `get_context -> send_direct_message / send_channel_message -> save_context` |
| Weather | `get_context -> get_weather -> send_channel_message` |
| Spot | `get_context -> search_spots / search_local_knowledge / search_combined -> send_channel_message` |
| Plan | `get_context -> get_driving_route -> send_channel_message` |

一句话：

**MCP 是把工具接进来，Function Calling 是让模型选对工具和参数。**

#### 13.3 为什么这个项目里 Function Calling 特别重要

因为这个项目不是“只要回答得像就行”，而是很多任务必须真的做对动作。

例如：

- 天气问题不能编天气，必须调 `get_weather`
- 路线问题不能编距离，必须调 `get_driving_route`
- Router 不能自己乱答专业问题，而要先 `get_context` 再 `send_direct_message`
- Spot 不能只会讲景点，还要决定是 `search_spots` 还是 `search_local_knowledge`

所以这个项目里，Function Calling 不是附属能力，而是主能力。

你甚至可以直接说：

> Router 这个任务本身，几乎就是一个 tool-calling 学习题。它不是在学“怎么把中文说漂亮”，而是在学“先读记忆、再分发、最后必要时补记忆”这条动作链。

#### 13.4 `function_call` 和 `observation` 分别是什么意思

这个问题和你的训练数据格式直接相关。

| 字段 | 在你项目里表示什么 |
| --- | --- |
| `function_call` | 模型预测的动作：调用哪个工具、传什么参数 |
| `observation` | 工具执行后的结果：返回值、回执、上下文读取结果 |

在你项目里很容易举例：

| 步骤 | 数据表达 |
| --- | --- |
| 读记忆 | `function_call: get_context({})` |
| 读到“暂无上下文记忆” | `observation: 暂无上下文记忆` |
| 转发给天气 Agent | `function_call: send_direct_message({...})` |
| 返回“消息已发送给weather_agent” | `observation: 消息已发送给weather_agent` |

也就是说：

- `function_call` 是“我要做什么”
- `observation` 是“做完之后系统告诉我发生了什么”

#### 13.5 为什么一开始是 ShareGPT 风格，后来又转成 `function_call + observation`

因为这两个格式分别服务于不同阶段。

| 阶段 | 更合适的格式 | 原因 |
| --- | --- | --- |
| 数据生成 / 记录动作轨迹 | `gpt + tool_calls` | 更像 OpenAI 聊天接口和真实轨迹 |
| LLaMA-Factory 训练 | `function_call + observation` | 更适合框架识别工具调用监督样本 |

所以你项目里的真实流程是：

1. 先生成 `ShareGPT` 风格原始样本  
   也就是 `human -> gpt(tool_calls) -> observation -> gpt(tool_calls) -> observation`

2. 再用 `convert_toolcalls_to_function_call.py` 转换  
   把 `gpt + tool_calls` 显式拆成 `function_call`

3. 这样 LLaMA-Factory 就更容易把它当作“动作轨迹监督样本”去学

这里最关键的一句可以直接背：

> 这个转换不是把业务逻辑改了，而是把同一条工具调用轨迹，从“聊天接口表达法”改成“训练框架更容易吃的表达法”。

## 1. 讲讲transformer大概内容原理。相比CNN\RNN突出的特点。另外说说现在还用得到Transformer吗？GPT是基于它么？ transformer为什么能处理不同长度的序列？

### 【面试当场如何回答】

- **通用答法**：Transformer 是基于 self-attention 的序列建模架构，用 attention 替代了 RNN 的串行递归结构。相比 RNN，它更容易并行、长距离依赖更容易建；相比 CNN，它不依赖固定局部卷积核，任意位置都能直接交互。现在当然还在用，GPT 就是 Decoder-only Transformer。它能处理不同长度序列，是因为 attention 和 mask 机制本身支持可变长度输入，只要位置编码和 padding mask 处理好就行。
- **结合我这个项目**：在我的 NLP 项目里，不管是阅读理解里用的 ERNIE / RoBERTa，还是现在大模型底座 Qwen，本质上都还是 Transformer 系列。也就是说，这不是过时结构，而是目前主流大模型和很多下游 NLP 模型的共同基础。

### 【详细解释】

#### 1.1 Transformer 核心

最标准的一句话：

**Transformer 用 self-attention 替代循环结构，让序列中任意位置都能直接交互。**

#### 1.2 Transformer vs CNN vs RNN

| 模型 | 优点 | 局限 |
| --- | --- | --- |
| CNN | 局部模式提取强、并行好 | 长距离依赖不自然 |
| RNN/LSTM | 适合顺序建模 | 串行计算，长依赖难 |
| Transformer | 并行强、长依赖好、扩展性强 | 长序列注意力成本高 |

#### 1.3 GPT 是不是基于它

是。  
GPT 属于 **Decoder-only Transformer**。

#### 1.4 为什么能处理不同长度序列

核心原因：

1. 输入是 token 序列，不要求固定长度
2. attention 本身支持变长
3. 用 padding mask 屏蔽补齐位
4. 用位置编码补顺序信息

## 3. 讲解一下attention机制？还有别的计算注意力的方式吗？attention中K,V维度不同该怎么办？padding，裁剪，维度变换？多头自注意力机制如何理解？

### 【面试当场如何回答】

- **通用答法**：Attention 的本质是用 `Q` 和 `K` 做匹配，再用匹配权重对 `V` 做加权汇总。标准公式是 `Attention(Q, K, V) = softmax(QK^T / sqrt(dk))V`。Padding 要靠 mask；Q 和 K 的最后一维必须能对齐做点积，V 的 value 维可以不同；多头自注意力就是让模型在不同子空间并行看不同关系。
- **结合我这个项目**：这套机制既是我 NLP 项目里 ERNIE / RoBERTa 的核心，也是我 Agent 项目底座模型 Qwen 的核心。LoRA 常加在 `q/k/v/o` 上，本质上就是在调注意力里“看什么、怎么匹配、取回什么、怎么汇总”的关键位置。

### 【详细解释】

标准公式：

`Q = XWQ`  
`K = XWK`  
`V = XWV`

`Attention(Q, K, V) = softmax(QK^T / sqrt(dk))V`

多头注意力：

`head_i = Attention(XWQ_i, XWK_i, XWV_i)`

#### 常见变体

| 类型 | 特点 |
| --- | --- |
| 加性注意力 | 用小网络打分 |
| 余弦注意力 | 用余弦相似度匹配 |
| 稀疏注意力 | 只算局部或稀疏连接 |
| 线性注意力 | 通过近似降低复杂度 |
| MQA / GQA | 共享或分组共享 K/V |

#### K、V 维度不同怎么办

要区分两件事：

1. **序列长度**
   - `len(Q)` 可以和 `len(K), len(V)` 不同
   - 但 `K` 和 `V` 的序列长度要一致
2. **特征维度**
   - `Q` 和 `K` 的最后一维要一致，才能点积
   - `V` 的 value 维可以不同

#### padding、裁剪、维度变换

- padding：用 mask 把 padding 位变成 `-inf`
- 裁剪：优先保留最近上下文和关键实体
- 维度变换：线性层把 `d_model` 投影到每个 head 的 `dk/dv`

## 4. resnet v1和v2的区别，resnet和normalization解决梯度消失，二者的区别，还有残差块放relu前还是后比较好？

### 【面试当场如何回答】

- **通用答法**：ResNet 的核心是残差连接，让网络学残差而不是直接学完整映射，解决深层退化和梯度传播困难。ResNet v1 是 `Conv-BN-ReLU` 的后激活结构，ResNet v2 是 `BN-ReLU-Conv` 的前激活结构。Normalization 稳定的是特征分布，残差连接改善的是信息和梯度的直通路径，两者不是替代关系。
- **结合我这个项目**：我当前主项目不是视觉模型，但这个问题我会按“结构差异 + 为什么更稳 + 和梯度消失的关系”来答，不会只背一个结论。

### 【详细解释】

| 维度 | ResNet v1 | ResNet v2 |
| --- | --- | --- |
| 激活位置 | 后激活 | 前激活 |
| 典型顺序 | `Conv-BN-ReLU` | `BN-ReLU-Conv` |
| shortcut 通过性 | 较弱 | 更顺滑 |
| 深层训练稳定性 | 好 | 通常更好 |

Residual 和 Normalization 的区别：

| 机制 | 主要解决什么 |
| --- | --- |
| 残差连接 | 让梯度和信息有更短路径 |
| Normalization | 稳定激活分布、提升训练稳定性 |

“ReLU 放前还是后”：

- 浅层网络差异不一定大
- 深层网络里，**前激活通常更稳**

## 5. 你提到了各自策略：adamW/lamb优化器、softmax交叉熵损失/focal loss损失函数、FGM/AWP对抗训练 你都知道这是什么原理，作用于模型哪些层吗？

### 【面试当场如何回答】

- **通用答法**：AdamW 和 Lamb 都是 Adam 系优化器，但 Lamb 额外做逐层自适应 trust ratio，更适合大 batch；softmax 交叉熵适合互斥分类，focal loss 适合强调困难样本；FGM 是在 embedding 上加扰动，AWP 是在权重上加扰动。
- **结合我这个项目**：在我的阅读理解项目里，最终实验上 AdamW 明显优于 Lamb，softmax 交叉熵优于 focal loss，FGM 优于 AWP。也就是说我不只是知道原理，还做过实际对比，最后选的是 `AdamW + softmax + FGM`。

### 【详细解释】

#### 5.1 AdamW vs Lamb

| 优化器 | 核心特点 | 适合什么 |
| --- | --- | --- |
| AdamW | Adam + 解耦权重衰减 | Transformer 微调、小中 batch |
| Lamb | AdamW + trust ratio | 大 batch 训练 |

项目里实验结果：

| 优化器 | F1 | EM |
| --- | ---: | ---: |
| AdamW | `87.34` | `74.45` |
| Lamb | `81.72` | `67.81` |

#### 5.2 softmax 交叉熵 vs focal loss

| 损失 | 适合什么场景 |
| --- | --- |
| softmax 交叉熵 | 互斥分类、位置竞争 |
| focal loss | 类别极不平衡、想强调难样本 |

项目里结果：

- softmax 明显优于 focal
- 原因是阅读理解答案位置本质上更像“互斥位置选择”

#### 5.3 FGM vs AWP

| 方法 | 扰动对象 | 作用层 |
| --- | --- | --- |
| FGM | Embedding 输入 | 嵌入层 |
| AWP | 模型权重 | 多个 weight 参数 |

项目里结果：

- FGM 比 AWP 稳定
- FGM 实现简单、超参更稳

## 8. F1分数和EM分数是什么？还有什么常见的评测指标？

### 【面试当场如何回答】

- **通用答法**：`EM` 是 Exact Match，要求预测答案和标准答案完全一致；`F1` 是精确率和召回率的调和平均，更关注部分重叠。阅读理解里一般同时报 `F1` 和 `EM`，因为一个看严格完全一致，一个看局部覆盖。
- **结合我这个项目**：在我的中医阅读理解项目里，`F1` 和 `EM` 都是核心指标；在 Agent 项目里，我又扩展到了 `Hit@K`、`MRR`、tool-calling exact match 这类更贴近系统行为的指标，所以我理解指标要和任务对齐，不能只会背 `F1/EM`。

### 【详细解释】

#### 8.1 EM

`EM = 1` 当且仅当预测答案和标准答案完全一致，否则为 `0`。

#### 8.2 F1

`F1 = 2 * precision * recall / (precision + recall)`

在抽取式问答里：

- `precision = overlap / pred`
- `recall = overlap / gold`

#### 8.3 常见指标

| 任务 | 常见指标 |
| --- | --- |
| 分类 | Accuracy、Precision、Recall、F1、AUC |
| 阅读理解 | F1、EM |
| 检索 | Hit@K、MRR、Recall@K、NDCG |
| 生成 | ROUGE、BLEU、BERTScore |
| Agent / 工具调用 | Tool-name accuracy、Argument exact match、Turn exact match、Success rate |

## 补充：这个项目有没有针对在线 badcase 做处理？这项工作好不好做？

### 【面试当场如何回答】

- **通用答法**：有，而且我觉得这类工作很重要。Agent 项目不能只看离线 benchmark，真正上线后经常会暴露出路由错、消息丢失、记忆不完整、工具边界不清这类在线 badcase。处理方式一般不是“重新训一下”这么简单，而是先定位出问题是在路由、Prompt、工具、记忆还是检索层，再做针对性修复。
- **结合我这个项目**：我这个项目其实连续做过多项面向在线 badcase 的专项修复，比如多 Agent 抢答和近似循环、`spot_agent` 工具已经调成功但前端没消息、城市级景点推荐默认先跑 RAG 导致慢响应、跨城行程记忆字段不够、酒店查询没有真实工具导致模型瞎编酒店名。这些都不是抽象讨论，而是我写成了单独修复文档并落到代码里的。
- **关于难不难做**：这项工作不算轻松，因为线上 badcase 往往是多模块耦合问题；但在我这个项目里还算相对方便做，原因是我把系统拆成了 `router + 专家 Agent + MCP 工具 + 记忆 + RAG`，边界比较清楚，所以很多问题可以定位到 1 到 3 个模块，不需要全系统推翻重来。

### 【详细解释】

#### 1. 这个项目里确实做过哪些在线 badcase 修复

你可以直接报出几个真实例子：

| badcase 类型 | 线上现象 | 真实修复思路 |
| --- | --- | --- |
| 多 Agent 抢答 / 互相触发 / 近似循环 | 一个问题多个 Agent 一起回，甚至彼此触发 | 做路由隔离：`travel_router` 只收公共消息，子 Agent 只收 direct message |
| 工具调了但前端没回复 | 日志里有答案，界面没消息 | 给 `spot_agent` 加 fallback runner，只在极窄条件下自动补发消息 |
| 城市级景点泛问法很慢 | “如皋市有什么好玩的”这类问题卡在 RAG | 改成 `POI-first`，城市级问题先查高德 POI，深问再补 RAG |
| 跨城行程记忆丢失 | “从故城县到如皋市”后再追问三天行程，系统只记住一个城市 | 扩展记忆 schema，新增 `origin_city / destination_city / departure_time / trip_days` |
| 酒店推荐出现幻觉 | 没有真实酒店工具时，模型直接编酒店名 | 给 `plan_agent` 接高德 POI 的 `search_hotels`，禁止无工具结果时瞎编 |

这些不是“口头说优化过”，而是仓库里都有对应修复文档：

- `2026-03-15_event_routing_fix.md`
- `2026-03-15_spot_agent_message_fallback_fix.md`
- `2026-03-15_spot_agent_poi_first_strategy_fix.md`
- `2026-03-15_trip_memory_schema_fix.md`
- `2026-03-15_plan_agent_hotel_poi_fix.md`

#### 2. 为什么这类在线 badcase 处理很有价值

因为离线指标再漂亮，也不代表真实交互就稳定。

这个项目里就很典型：

- `router` 离线 benchmark 可以证明分发能力提升
- 但线上仍然可能出现“分发是对的，消息没真正发出去”
- RAG 离线命中率可以看召回
- 但线上仍可能因为触发策略不合理，导致城市级问题先卡在 RAG 上
- Prompt 看起来没问题
- 但多轮对话里会暴露记忆 schema 不够表达跨城任务

所以在线 badcase 的意义是：

- 它能暴露“离线评测看不到的系统级问题”
- 它能帮助你把问题从“模型不稳定”缩小到某个具体模块
- 它更能体现工程能力，而不是只会调模型

#### 3. 这项工作好做吗？

最稳妥的回答不是“好做”，也不是“特别难”，而是区分情况。

| 维度 | 结论 | 为什么 |
| --- | --- | --- |
| 发现问题 | 不算太难 | 真实聊天一跑就能暴露，日志和前端现象也比较直接 |
| 精准定位 | 中等偏难 | 线上 badcase 往往跨路由、Prompt、工具、记忆、RAG 多层 |
| 做小修复 | 相对容易 | 如果系统边界清楚，很多问题能收敛到 YAML、runner 或工具层 |
| 做长期治理 | 不轻松 | 需要回归测试、日志、评测集和版本化修复文档支持 |

结合你这个项目，我会这样说：

- **相对方便的地方**：
  - 架构拆得比较清楚，`router`、`weather`、`spot`、`plan` 职责分明

  - 当前自动化回归和端到端对话回放还不够强

  - 如果没有专项修复文档，时间一长很容易忘记为什么这么改

#### 4. 你这个项目里为什么这项工作“能做得动”

这是面试里很容易加分的一点，因为你不是只说“我修过 bug”，而是能解释为什么修得动。

我这个项目之所以还比较适合做在线 badcase 修复，主要因为：
