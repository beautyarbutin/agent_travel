# 2026-03-15 RAG 审计、评测与改进方案

## 1. 这次成功对话里，Agent 间的数据是怎么流动的

这轮成功对话的链路已经比较清晰，采用的是“`travel_router` 单入口分发 + 专家 Agent 执行”的结构。

### 1.1 天气问题

用户先问“故城县天气如何”。实际流转是：

1. 用户消息进入公共频道 `general`
2. `travel_router` 识别为天气意图
3. `travel_router` 通过 `send_direct_message` 把任务发给 `weather_agent`
4. `weather_agent` 调用 `get_weather(city="故城县")`
5. `weather_agent` 再把结果发回 `general`
6. `travel_router` 看到这条来自 `weather_agent` 的频道消息后直接 `finish`，避免循环回复

这一段没有用 RAG，只用了天气 MCP 工具。

### 1.2 路线问题

用户接着问“从那里到如皋市，怎么走”。这轮里：

1. `travel_router` 先读取记忆
2. 结合上文把“那里”消解成 `故城县`
3. `travel_router` 保存跨城槽位：
   - `origin_city="故城县"`
   - `destination_city="如皋市"`
4. `travel_router` 通过 `send_direct_message` 把任务发给 `plan_agent`
5. `plan_agent` 调用 `get_driving_route(origin="故城县", destination="如皋市")`
6. `plan_agent` 调用 `send_channel_message(channel="general", text=...)` 发回路线结果
7. `travel_router` 收到这条专家消息后忽略，不再二次接管

这一段也没有用 RAG，只用了高德路线规划接口。

### 1.3 景点问题

用户再问“如皋市有什么好玩的”。这轮里：

1. `travel_router` 识别为景点意图
2. `travel_router` 把问题 direct message 给 `spot_agent`
3. `spot_agent` 调用 `search_spots(query="如皋市有什么好玩的地方推荐吗")`
4. 这一轮没有调用 `search_local_knowledge`
5. 模型虽然生成了景点推荐文本，但没有自己调用 `send_channel_message`
6. 因为我给 `spot_agent` 加了专属 fallback runner，所以系统兜底把这段文本补发到了 `general`

所以这轮前端能看到 `spot_agent` 的回复，但它是“查询成功 + fallback 补发成功”，不是“模型自己稳定完成了消息发送”。

### 1.4 这轮有没有用 RAG

**没有。**

这轮成功对话中：

- 天气：只调用了 `get_weather`
- 路线：只调用了 `get_driving_route`
- 景点：只调用了 `search_spots`

没有出现这两个工具：

- `search_local_knowledge`
- `search_combined`

这和现在的 `spot_agent` 策略一致：城市级泛问法优先走高德 POI，不默认先跑本地 RAG。

## 2. 你这个项目的 Hybrid RAG 现在到底是怎么实现的

### 2.1 真实入口

真正给 Agent 暴露工具的是 `mcp_server.py`：

- `search_spots(query)`：高德 POI 实时检索
- `search_local_knowledge(query)`：调用本地 RAG
- `search_combined(query)`：本地 RAG + 高德 POI 融合

其中：

- `search_local_knowledge()` 会转到 `tools/spot_tools.py` 里的 `search_knowledge()`
- `search_combined()` 会转到 `tools/spot_tools.py` 里的 `search_combined()`

### 2.2 本地 RAG 的实现组成

本地 RAG 主要在 `tools/spot_tools.py`，它有 3 层：

1. **FAISS 向量检索**
   - 向量索引文件：`storage/faiss_index.bin`
   - 文档文件：`storage/doecment.json`
   - Embedding 模型：`BAAI/bge-small-zh-v1.5`

2. **关键词打分**
   - 文件里函数名叫 `_bm25_score()`
   - 但严格说它**不是标准 BM25 公式**
   - 它更像一个“BM25 风格的手工词法打分器”

3. **高德地图 POI API**
   - 用来查实时景点地址、城市范围、电话等外部数据

### 2.3 当前 Hybrid RAG 的真实工作方式

#### A. `search_knowledge(query)`

只走本地知识库：

1. 加载 `doecment.json`
2. 如果有 FAISS 索引，就做一次向量召回 Top-5
3. 对全库做一次 `_bm25_score(query, spot)` 打分
4. 把向量分和关键词分加权求和
5. 若来源是 `独家手写攻略` 且有关键词命中，再额外加 `HANDWRITTEN_BONUS = 35.0`
6. 输出 Top-K

#### B. `search_spots(query)`

只走高德：

1. 从自然语言里抽地名提示
2. 调高德 `place/text`
3. 返回景点名、地址、行政区划、电话

#### C. `search_combined(query)`

这是你项目里最像“Hybrid RAG”的入口：

1. 先跑本地 `search_knowledge(query)` 拿深度攻略
2. 再跑高德 `search_spots(query)` 拿实时 POI
3. 最终把“两层信息”拼到一个回答里

### 2.4 一个很重要的诚实说法

如果面试官深问，“你这是不是标准 BM25 + FAISS + 高德 API 的 Hybrid RAG？”

更严谨的回答应该是：

> 我现在的实现是 **FAISS 向量检索 + BM25-like 词法打分 + 高德 POI 外部检索** 的混合方案。  
> 其中词法部分目前是我自己写的启发式打分器，不是标准 BM25 公式，这是后续可优化点。

这个回答比直接说“我用了 BM25”更稳，更不容易被继续追问时问穿。

## 4. 为什么上次评测 Hit@ 都没超过 50%

### 4.1 现有评测结果

你仓库里最新一版现成报告是 `reports/rag_eval_20260313_153914/report.md`，结果如下：

#### 严格 Doc-ID

- `Hit@1 = 35.9%`
- `Hit@3 = 36.9%`
- `Hit@5 = 36.9%`
- `MRR = 0.3636`

#### 宽松实体级

- `Hit@1 = 42.9%`
- `Hit@3 = 43.9%`
- `Hit@5 = 43.9%`
- `MRR = 0.4343`

### 4.2 不是单纯“数据烂”，是 4 个问题叠加

#### 问题 1：数据分布严重失衡

现在 95% 以上都是 `Kaggle/去哪儿网`，而真正高质量的 `独家手写攻略` 只有 347 条。

这意味着：

- 检索库里“广度数据”很多
- 但“深度数据”很少
- 一旦问到区县级、冷门市县级、细粒度玩法，命中概率就会快速掉下去

#### 问题 2：县级/区县级覆盖太差

我做了几个简单覆盖检查：

- `如皋`：只命中 2 条
- `故城县`：只命中 2 条
- `衡水湖`：只命中 1 条

这会直接导致：

- 你 demo 的“故城县、如皋市、庆云县”这类问题非常依赖少量手写攻略
- 一旦没有对应手写条目，RAG 很难稳住

#### 问题 3：你这个 `_bm25_score()` 不是标准 BM25，而且有误召回风险

当前词法打分包含这些规则：

- 城市命中加分
- 区县命中加分
- 景点名命中加分
- tag 命中加分
- 再做一个“字符出现在 content 里”的重叠加分

这里最大的问题是最后这条：

> 只要 query 中的单个字出现在内容里，就会加一点分。

这会导致大量“语义并不相干，但碰巧有相同字”的文档也得分。

比如我查 `故宫博物院` 时，库里出现了 **11 条**文本命中，其中混进了不少并非故宫的景点条目。

#### 问题 4：`HANDWRITTEN_BONUS = 35.0` 太大

这条 bonus 的初衷是对的：保护你的私域核心资产。

但副作用是：

- 只要有一点点词法碰撞，手写攻略就很容易长期压过 Kaggle 公域条目
- 这也是为什么很多严格 Doc-ID 评测会 miss，但宽松实体级会好一些

简单说就是：

- “实体大方向”有时找对了
- 但“你指定的那条文档 ID”没有排到前面

### 4.3 评测里最直观的信号

报告里分类表现很典型：

- `手写`：`Hit@1 = 91.7%`
- `Kaggle`：严格 `Hit@1 = 0%`
- `城市级`：宽松 `Hit@1 = 86.8%`
- `精确景点`：严格 `Hit@1 = 23.8%`

这说明：

1. 你的手写攻略这层其实很强
2. 城市级泛问法也还行
3. 真正差的是“精确景点名 -> 精确文档命中”这件事
4. 所以问题不是“RAG 完全没用”，而是它当前更像“粗召回 + 手写优先”，不是“高精度实体检索”

## 7. RAG 评测指标是什么，公式是什么

你项目里当前主要在用的是：

- `Hit@K`
- `MRR`

### 7.1 Hit@K

含义：

> 对于一批查询，如果“第一个相关结果”出现在前 K 个返回结果里，就算命中。

公式：

```text
Hit@K = 命中查询数 / 总查询数
```

更展开地写：

```text
Hit@K = (1 / N) * Σ I(rank_i <= K)
```

其中：

- `N`：查询总数
- `rank_i`：第 i 个查询的首个相关结果排名
- `I(...)`：指示函数，成立为 1，否则为 0

### 7.2 MRR

全称：

- `Mean Reciprocal Rank`

含义：

> 关注“第一个相关结果排得有多靠前”。

公式：

```text
MRR = (1 / N) * Σ (1 / rank_i)
```

如果某条 query 没命中，则该项记为 0。

举例：

- 第 1 名命中，贡献 `1.0`
- 第 2 名命中，贡献 `0.5`
- 第 5 名命中，贡献 `0.2`
- 完全没命中，贡献 `0`

### 7.3 你项目里还有“严格”和“宽松”两套评估

#### 严格 Doc-ID

要求：

- 必须返回你期望的那条文档 ID

优点：

- 很严格，适合看“精确检索能力”

缺点：

- 对“同实体但不同来源条目”不友好

#### 宽松实体级

要求：

- 只要是“同城市 + 同景点实体”的跨来源条目，也算命中

优点：

- 更接近真实用户体验

缺点：

- 指标会比严格版更高

## 8. 面试时可能会问什么，以及你怎么答

### Q1：你的 RAG 为什么 Hit@1 只有三四十？

答：

> 这个结果主要有三方面原因。第一，知识库分布极不均衡，95% 以上是公域的景点介绍，真正高质量的私域攻略很少。第二，县级和区县级覆盖不足，比如如皋、故城县这类城市条目非常少。第三，当前词法检索部分是我自己写的 BM25-like 打分器，不是标准 BM25，因此在精确景点检索上还存在误召回。  
> 但从分类结果看，私域手写攻略 Hit@1 已经到 91.7%，说明问题不在于 RAG 思路失效，而在于数据结构和排序策略还需要进一步工程化。

### Q2：你这个项目为什么叫 Hybrid RAG？

答：

> 因为我不是只做本地向量检索。我把系统拆成了三路：  
> 第一条是 FAISS 向量召回，负责语义相近；  
> 第二条是词法打分，负责城市名、景点名、标签词这些精确约束；  
> 第三条是高德 POI 实时检索，负责地址、电话、实时景点候选。  
> 这三路在不同问题下分工协作，所以我把它定义为 Hybrid RAG。

### Q3：为什么这次成功回答“如皋市有什么好玩的”时没有用 RAG？

答：

> 这是我有意做的策略调整。城市级泛推荐问题优先走高德 POI，因为它更快、更稳，而且能先拿到结构化候选景点。  
> 只有当用户继续追问某个景点的具体玩法、避坑、拍照机位、美食推荐时，我才补充本地 RAG。  
> 也就是说，Hybrid RAG 并不是每一轮都强制三路全开，而是按问题类型动态选择最合适的那一路。

### Q4：你现在最大的 RAG 技术债是什么？

答：

> 最大的技术债有两个。第一，当前 `_bm25_score()` 不是标准 BM25，而是手工启发式词法打分；第二，知识库没有按“POI 元数据”和“深度攻略文本”分层，导致不同质量的数据被混在一个索引里。  
> 我下一步会做的就是把主库拆分，并且把词法检索替换成标准 BM25 或中文倒排检索方案。

### Q5：如果让你继续改，你会先改哪三件事？

答：

> 第一，拆库：把深度攻略、POI 元数据、训练评测数据分开。  
> 第二，检索升级：把现在的 BM25-like 打分替换成标准 BM25，并加城市过滤。  
> 第三，数据补全：重点补如皋、故城、庆云这类下沉市场区县级攻略，而不是继续盲目加全国通用景点介绍。

### Q6：为什么宽松实体级比严格 Doc-ID 好很多？

答：

> 因为系统经常能找到对的景点实体，但不一定是我评测集中指定的那条来源文档。  
> 比如问“故宫博物院怎么玩”，它可能先返回我手写的“故宫博物院与景山公园”组合攻略，而不是 Kaggle 那条单独的“故宫博物院”文档。  
> 从用户体验看这未必是错，但从严格 Doc-ID 看就算 miss。

## 10. 最终判断

### 10.1 这套 RAG 值不值得继续做

**值得。**

理由：

- 你的 `独家手写攻略` 已经证明私域攻略很有价值
- 当前城市级泛推荐 + 高德 API 已经能稳定工作
- 问题主要不是“路线错了”，而是“库结构、数据层次、排序逻辑还不够工程化”

### 10.2 你现在最该做的事

不是继续盲目堆条数，而是：

1. 先把检索架构讲清楚
2. 诚实承认当前 BM25 还是 BM25-like
3. 把深度攻略与 POI 元数据分层
4. 再补高质量的区县级攻略数据

### 10.3 一句适合面试的总结

> 我这个项目的 RAG 并不是单纯“把文本塞进向量库”那么简单，而是把本地攻略知识、高德实时 POI、路线 API 和多 Agent 调度结合起来做了一个面向旅游场景的 Hybrid RAG。  
> 当前版本已经跑通了天气、路线和景点推荐，但我也识别出 RAG 里还存在数据结构失衡、县级覆盖不足和 BM25-like 排序不够标准的问题。下一步我会继续把它从“能跑”打磨到“可解释、可评测、可扩展”。
