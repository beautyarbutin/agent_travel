# 李树岩6.0.pptx 评价与复试面试问答

## 1. 总体评价

一句话判断：这是一份“内容硬、项目真、技术细节多”的 PPT，但当前版本更像毕业设计汇报或项目答辩，不够像研究生复试场景下的高效个人陈述。

如果按“复试 PPT”标准打分，我会给出下面这个评价：

| 维度 | 评价 |
| --- | --- |
| 内容实力 | 8.5/10 |
| 技术含量 | 8.5/10 |
| 研究潜力呈现 | 7/10 |
| 现场表达友好度 | 6/10 |
| 结构取舍 | 6/10 |
| 综合 | 7.5/10 |

核心结论：

- 优点很明显：做过真实项目，有工程深度，有实验数据，也能主动分析 badcase 和后续改进。
- 主要问题也很明显：页数过多、信息过满、技术细节偏“实现汇报”，而“为什么适合读研、研究兴趣是什么、和目标方向如何契合”讲得不够集中。
- 如果直接拿去复试，老师会觉得“这个学生做过事”，但也很容易产生两个风险：一是超时，二是老师抓住局部实现细节追问，反而削弱你对整体能力和研究潜力的呈现。

## 2. 这份 PPT 的优点

### 2.1 项目一是整套材料的最大亮点

- 多智能体旅游助手这个项目完整度较高，不是简单的“调 API 做个 demo”，而是覆盖了路由、工具调用、LoRA 微调、Hybrid RAG、记忆机制、MCP 工具协议等多个层面。
- 你不只展示“做了什么”，还展示了“怎么迭代出来的”，例如从广播抢答到中心化路由、从默认先跑 RAG 到 POI-first、从坏例分析到后续优化，这一点很加分。
- 有量化结果支撑，例如 router 链路成功率提升、spot_lora 的指标变化、RAG 离线评测等，说明不是纯口头描述。

### 2.2 项目二能体现基础能力

- 机器阅读理解项目说明你不是只会搭 Agent，也有较规范的深度学习训练和实验对比经验。
- 你展示了预训练模型对比、损失函数、数据增强、对抗训练、消融实验、部署，这有助于证明你具备基础研究训练能力。

### 2.3 你有“问题意识”

- PPT 不是只讲成功案例，而是专门讲了 badcase、难点解决、未来改进，这比单纯堆结果更像一个能继续做研究的人。
- 老师通常会看重“你是否知道自己的系统哪里还不够”，这一点你是有体现的。

### 2.4 整体风格基本统一

- 主体大多数页面采用统一的蓝白配色和页眉样式，正式感是够的。
- 封面、目录、章节页、项目主线都比较完整，说明你有基本的汇报意识。

## 3. 主要不足

### 3.1 结构上的问题：太长，主次不够聚焦

- 全套 45 页，对复试来说明显偏长。若复试陈述只有 5 到 8 分钟，这个页数几乎必然超时。
- 项目一占比过大，个人介绍、报考动机、研究兴趣、未来规划反而偏弱。老师更关心“你为什么适合读研”，而不只是“你做了一个很长的项目”。
- 多个章节过渡页和同主题页面可以合并，例如章节大页较多，项目一内部也有若干可压缩页面。

最直接的问题不是“内容多”，而是“信息密度分配不合理”：

- 老师真正会记住的是 3 到 5 个关键信息。
- 你现在给了太多细节，反而容易淹没最想让老师记住的亮点。

### 3.2 内容上的问题：偏实现汇报，不够像复试陈述

- 项目一里有不少页面在讲 prompt 规则、agent 间 direct message、字段名、工具名、memory 字段等，这些内容对技术老师有价值，但对复试场景来说过细了。
- 很多页面更像“我怎么把系统做出来”，但“这个问题为什么重要、我提出了什么方法、效果提升说明了什么”总结得不够锋利。
- 项目二部分虽然有实验，但更像毕业论文摘要，个人贡献、研究启发、方法选择理由可以说得更集中。

复试更好的表达方式应当是：

- 先讲问题。
- 再讲方法。
- 然后讲你自己的贡献。
- 最后讲结果和反思。

而不是把运行细节平铺展开。

### 3.3 页面表现上的问题：有些页太满，有些页信息价值偏低

下面这些页面类型，现场展示时负担会比较重：

- 个人介绍页：信息偏满，联系方式、邮箱、成绩、课程、照片、分数表同时出现，视觉重点不够集中。
- 项目一中多页“长段文字 + 小图/流程图”组合：如 LoRA、Prompt 优化、Badcase、未来改进等页面，文字较多，老师很难在短时间内抓住重点。
- 聊天截图类页面：如最终效果、难点解决页，截图较长，字体偏小，远距离看不清。
- 表格截图类页面：如预训练模型对比、消融实验等页，如果是投屏或线上面试，细节可读性一般。
- 奖项与英语证明页：证书图片较多，但信息效率不高，还暴露了较多个人敏感信息。

### 3.4 版式一致性还不够

- 大多数页面是统一蓝白风格，但项目二中部分表格页、背景图页明显像直接从论文或原始材料里截出来的，和前面的页面风格不完全一致。
- 有些页面标题很规范，有些页面更像文档截图，整体会给人“后半部分是拼接进去的”感觉。
- 对复试 PPT 来说，统一性会影响老师对“准备是否充分”的第一印象。

### 3.5 复试定位还不够强

当前 PPT 最大的缺项不是技术，而是“读研匹配度”的表达不够强，包括：

- 为什么想继续读研。
- 为什么选这个方向。
- 为什么和目标学院、目标导师契合。
- 你读研后想解决什么问题。

最后的未来规划页过于概括，像口号，不够具体，难以支撑“研究潜力”这一判断。

### 3.6 存在隐私与答辩风险

- 个人介绍页公开了手机号和邮箱。
- 英语证明页中包含证件号、准考证号等敏感信息。
- 若这份 PPT 用于线上面试、共享屏幕、材料归档或发给老师，建议立刻打码处理。

### 3.7 具体到页的修改建议

- 第 4 页个人简介：信息过满，建议删去手机号和完整邮箱，只保留学校、专业、排名、英语、核心标签即可。
- 第 7 页项目定位：内容是好的，但右侧文字仍偏多，建议收敛成“问题、方法、结果”三行式表达。
- 第 23 页 Prompt 优化：这是典型的信息过载页，适合拆成“流程图 + 三条约束结论”，不要同时讲 4 个 step 和一大块补充说明。
- 第 25 至 27 页难点解决与最终效果：截图较长，远看不清，建议只截关键局部，并用红框或箭头标出改进前后差异。
- 第 34 至 36 页项目二实验结果：部分页面风格与前文不一致，建议把表格重绘成统一样式，至少保证标题、背景和字体系统一致。
- 第 42 页英语证明：与第 4 页信息重复，还存在隐私风险，建议直接删除。
- 第 44 页未来规划：表述偏空，建议改成“研究兴趣 + 拟解决问题 + 读研目标”三点式。

## 4. 最值得优先修改的地方

如果时间有限，我建议只优先改下面 6 点，收益最高。

### 4.1 把总页数压到 12 到 18 页

推荐保留逻辑：

1. 封面
2. 个人简介
3. 本科经历与核心优势
4. 项目一：问题与目标
5. 项目一：方法与架构
6. 项目一：实验结果与改进
7. 项目一：你的贡献与反思
8. 项目二：研究问题与方法
9. 项目二：结果与结论
10. 奖项与综合素质
11. 读研目标与研究兴趣
12. 致谢

如果还想更稳，可以扩展到 14 到 16 页，但不建议继续维持 45 页。

### 4.2 项目一必须压缩，但不能削弱亮点

项目一现在是最强内容，但讲法应从“展开叙述”改为“打关键点”：

- 只保留一页问题背景。
- 一页架构图。
- 一页核心优化点。
- 一页实验结果。
- 一页 badcase 与改进。

也就是说，项目一最好控制在 5 页左右，而不是二十多页。

### 4.3 把“代码词汇”翻译成“答辩词汇”

比如下面这些词，不要在讲述时停留在实现命名层面：

- `travel_router`
- `spot_agent`
- `get_context`
- `save_context`
- `direct message`

更好的口头表达方式是：

- 统一路由模块
- 景点问答模块
- 上下文读取机制
- 记忆写回机制
- 定向分发机制

老师关心的是方法思想，不是你变量名起得多细。

### 4.4 项目二要补“个人贡献”和“方法选择理由”

项目二容易被问：

- 为什么选 ERNIE 而不是别的模型？
- 为什么 5% 掩码有效？
- FGM 为什么能提升鲁棒性？
- 你自己做了哪些工作，而不是照着论文复现？

因此建议在项目二里明确补一句：

- 我的主要工作包括：数据处理、滑动窗口实现、训练策略组合设计、消融实验、模型部署与可视化验证。

### 4.5 奖项页做减法

建议：

- 保留最能说明问题的 3 到 4 项即可。
- 证书不用整页展示，可以用“奖项名称 + 年份 + 级别”列表概括。
- 英语成绩不用再单独放证明截图，因为个人介绍页里已经写过。

### 4.6 未来规划页要写具体

建议把“未来规划”改成“拟开展研究兴趣与读研目标”，例如：

- 希望继续在大模型 Agent、RAG、工具调用可靠性方向深入。
- 想重点关注多轮上下文记忆、检索准确率和系统鲁棒性问题。
- 入学后先补足机器学习、NLP、信息检索等理论基础，再结合导师方向开展实验。

这样比“夯实基础、产学结合、Keep studying”更像复试回答。

## 5. 老师可能最在意的追问点

你这套材料最可能被老师追问的，不一定是“你会不会”，而是“你说的提升是否站得住”。重点有四类：

- 指标是否可信：成功率怎么定义，样本量多大，是否有 held-out 测试。
- 贡献是否清楚：哪些工作是你自己完成的，哪些是基于现有框架。
- 方案是否合理：为什么选 LoRA、为什么要多 Agent、为什么要 Hybrid RAG。
- 研究潜力是否明确：这个项目下一步可以形成什么研究问题，而不只是继续堆工程功能。

## 6. 复试面试问答题

下面的问题按复试高频逻辑整理，并给出可参考的简洁回答。回答不要背诵，建议按“结论先行 + 两到三句解释”来讲。

### 6.1 自我介绍与读研动机

**Q1：请做一个简短的自我介绍。**

**参考回答：**
我叫李树岩，本科就读于燕山大学软件工程专业，英语六级 574 分，排名 46/224。本科阶段我主要围绕两个方向积累能力：一是大模型 Agent 与 RAG 系统开发，二是机器阅读理解方向的深度学习实验。我希望在研究生阶段继续在 NLP 和大模型应用方向深入，把工程实践经验和更系统的理论训练结合起来。

**Q2：你为什么想读研？**

**参考回答：**
本科阶段我做项目时越来越明显地感受到，工程实现可以把系统搭起来，但很多关键问题，比如模型为何失效、检索为何不稳定、如何设计更合理的评测，需要更扎实的理论支持。所以我想通过读研补足方法论和研究训练，把“会做系统”提升为“能提出问题并做系统性验证”。

**Q3：你为什么选择这个研究方向？**

**参考回答：**
我对 NLP 和大模型系统方向感兴趣，主要是因为它既有模型层面的研究问题，也有系统层面的落地价值。本科项目里我接触了多智能体、RAG、工具调用、鲁棒性训练等内容，越做越觉得这个方向既有挑战，也适合我继续深挖。

**Q4：你觉得自己最大的优势是什么？**

**参考回答：**
我的优势是能把问题从“想法”推进到“可运行系统”，同时也愿意做实验和 badcase 分析。我不满足于只把功能跑通，而是会继续关注指标、失败样本和后续优化方向。

### 6.2 项目一：基于 OpenAgents 的多智能体旅游助手

**Q5：为什么这个项目要做成多智能体，而不是单智能体？**

**参考回答：**
因为旅游场景里任务类型比较杂，包含天气、景点、路线、行程等，不同任务对工具和回答方式的要求不同。单智能体虽然实现简单，但职责容易混杂；拆分成多智能体后，可以把任务边界和工具边界分清，系统更容易扩展和优化。

**Q6：你们为什么要从广播抢答改成中心化路由？**

**参考回答：**
广播监听的问题是容易出现多 Agent 同时响应、误触发和回环，消息流不稳定。改成中心化路由后，只有 router 负责接收用户消息，再根据意图定向分发给对应 Agent，这样职责更清晰，系统行为也更可控。

**Q7：你在这个项目中的主要贡献是什么？**

**参考回答：**
我主要参与了多智能体整体方案设计、router 和 spot 两个模块的微调数据构造、Prompt 规则优化、RAG 检索流程设计，以及 badcase 分析和评测整理。我的重点工作不是单点功能，而是把系统流程逐步稳定下来。

**Q8：为什么只对 router 和 spot 做 LoRA 微调？**

**参考回答：**
因为从系统瓶颈看，router 决定消息是否正确分发，spot 决定旅游问答质量，它们对整体体验影响最大。同时受显存限制，不可能同时为所有模块单独训练模型，所以优先优化最关键的两个模块。

**Q9：你提到 router 的完整链路成功率从 20.59% 提升到 85.29%，这个指标是怎么定义的？**

**参考回答：**
这个指标不是只看“意图是否识别对”，而是看一整条链路是否成功，包括是否先读取记忆、是否正确识别任务、是否分发到正确 Agent、参数是否补全、以及是否正确保存上下文。我们在 held-out 的 tool-calling 样本上做了评估，因此它反映的是端到端执行质量。

**Q10：Prompt 优化和 LoRA 微调各自起什么作用？**

**参考回答：**
Prompt 更像运行时规则设计，规定模型应该先读记忆、再做指代消解、再分发任务；LoRA 的作用是把这些规则进一步内化，让模型在复杂场景下更稳定地执行这些动作。简单说，Prompt 负责定义规则，LoRA 负责增强执行规则的稳定性。

**Q11：为什么你们采用 Hybrid RAG，而不是只做向量检索？**

**参考回答：**
旅游问题里既有语义相近但表述不一致的情况，也有景点名、地名这类关键词精确匹配需求。只用向量检索容易丢掉关键词约束，只用 BM25 又可能缺少语义泛化能力，所以把 bge 向量检索和 BM25 结合起来，更适合这个场景。

**Q12：为什么后来改成 POI-first，而不是默认先查 RAG？**

**参考回答：**
因为城市级泛问题其实用地图 POI 就能较快得到高质量结果，如果一上来先跑 RAG，响应会变慢，而且未必带来额外收益。后续我们把流程改成 POI-first，只在需要攻略、避坑或深度知识时再调本地 RAG，这样速度和效果更平衡。

**Q13：这个项目目前最大的短板是什么？**

**参考回答：**
我认为最大短板有三个：一是 RAG 还缺少 rerank，精确检索能力不够稳定；二是记忆机制还没有做到严格的用户隔离和会话隔离；三是 weather 和 plan 模块还没有针对性模型，整体还有进一步细化优化空间。

**Q14：MCP 在这个项目里解决了什么问题？**

**参考回答：**
MCP 主要解决的是工具接入方式不统一的问题。它相当于把天气、景点、路线、记忆这些能力标准化暴露出来，让不同 Agent 可以通过统一协议调用工具，从而降低耦合度，也便于后续扩展。

**Q15：如果让你继续把这个项目做成研究方向，你会怎么展开？**

**参考回答：**
我会重点往三个问题展开：第一，如何提高多轮上下文记忆的稳定性；第二，如何通过 rerank 和数据质量控制提升 RAG 命中率；第三，如何建立更规范的 Agent 系统评测指标，而不只是看单次回答质量。

### 6.3 项目二：基于深度学习的机器阅读理解模型研究

**Q16：这个项目的研究目标是什么？**

**参考回答：**
目标是在 DuReader-robust 数据集上比较多种预训练模型和训练策略，找出更适合中文机器阅读理解任务的方案。我的重点不是提出全新模型，而是通过系统对比实验找到更优组合。

**Q17：为什么你最后选择了 ERNIE-3.0-xbase-zh？**

**参考回答：**
因为它在中文语义理解任务上有较强基础能力，在我们的实验里也取得了更好的 F1 和 EM 表现。相比单纯追求更大模型，这个模型在效果和训练成本之间更平衡，适合本科阶段的实验条件。

**Q18：为什么要使用 doc_stride？**

**参考回答：**
DuReader-robust 里有不少长文本，如果直接截断，会把答案所在位置截掉。使用滑动窗口和 doc_stride 可以把长文分块送入模型，同时保留上下文覆盖，从而减少长文本场景下的信息丢失。

**Q19：5% 掩码增强为什么会有效？**

**参考回答：**
我的理解是，它能降低模型对局部高频词和表面模式的过度依赖，迫使模型更多利用上下文信息完成预测。因此在有噪声或表述变化的情况下，模型的泛化能力会更稳一些。

**Q20：FGM 对抗训练的作用是什么？**

**参考回答：**
FGM 会在嵌入层加入一个朝损失上升方向的小扰动，相当于在训练时主动制造更难的样本。这样模型会学到更稳健的表示，对输入中的微小变化不那么敏感，因此鲁棒性会提升。

**Q21：EM 和 F1 的区别是什么？**

**参考回答：**
EM 看的是预测答案和标准答案是否完全一致，是更严格的指标；F1 则综合考虑答案片段重叠的精确率和召回率，更能反映“答得接近但不完全一样”的情况。所以 MRC 任务里通常会同时看这两个指标。

**Q22：项目二里你个人最有价值的收获是什么？**

**参考回答：**
最大的收获是我学会了用更规范的实验方式看问题，不只是看最终分数，而是会比较模型、比较训练策略、做消融实验、分析指标差异。这种实验思维对后续做研究非常重要。
